# Agent 验证策略

## 完成判定原则（强制）

1. 完成判定只能来自脚本验证结果，不能来自 Agent 主观判断。
2. Rule 只能定义约束，不能替代验证执行。
3. 任务必须以“验收标准 -> 脚本验证”方式闭环。

## 总验证入口（统一执行顺序）

每次提交前按以下顺序执行：

1. 静态与工程质量
	- `pnpm lint`
	- `pnpm typecheck`
2. 基础回归
	- `pnpm test`
3. 任务特定脚本（按改动范围选择）
	- 日报链路：`pnpm run-daily -- --date <date> --dry-run`
	- 周报链路：`pnpm run-weekly -- --date <date> --dry-run`
	- 评分链路：`pnpm score -- --input <normalized-file>`
	- 知识库链路：`pnpm build-kb -- --since <date>`

只要有一项失败，就不能标记任务完成。

## 验收标准到脚本映射要求

每个任务都必须在执行前建立映射清单，至少包含：

1. 验收条目（来自用户请求、spec、exec-plan）。
2. 对应脚本命令。
3. 通过标准（例如退出码为 0、关键输出存在、无异常写入）。
4. 执行证据（命令、关键输出、失败日志摘要）。

## 重点测试要求

1. Signal parser 必须有 snapshot 输入。
2. Score component 必须覆盖有证据和无证据两种路径。
3. rules-only 模式必须有回归测试。
4. dry-run 必须证明不会写持久状态。
5. weekly report 必须断言输出趋势短语，而不是只输出项目清单。

## 无法运行验证时（强制披露）

必须说明：

1. 哪个脚本无法执行，以及具体原因。
2. 是否尝试替代验证方式。
3. 仍未覆盖的验收标准。
4. 当前残余风险与影响范围。
