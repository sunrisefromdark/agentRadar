## 一句话冻结

本次不是“把当前页面修得更好看一点”，而是把 `Trend Radar` 从“能看信息的可视化页面”升级为“有品牌感、有科技感、有判断效率、有研究氛围的趋势情报工作台”。

## 1. 背景与问题定义

### 1.1 当前产品现状

当前仓库已经具备浏览器版可视控制台，拥有以下基础能力：

- 五个一级视图：`Overview`、`Projects`、`Weekly`、`Run Health`、`Knowledge Base`
- 本地启动的 Web 浏览器入口
- 明确的 `artifact-first` 消费边界
- 详情侧栏、双语切换、主题切换、评分雷达与 KB 阅读基础能力

这说明系统已经跨过了“有没有 Web 界面”的阶段，当前真正的问题不再是功能缺失，而是产品外观与体验层次还没有达到一个成熟前沿开源产品应有的完成度。

### 1.2 当前 UI 的核心问题

虽然现有前端已经有过一轮样式升级，但整体仍存在以下结构性问题：

1. 页面气质仍偏“可用的后台/看板”，不是“趋势情报工作台”。
2. 整体布局仍偏均质卡片堆叠，信息密度与视觉层次没有拉开。
3. 五个一级视图共享同一套基础样式，但缺少各自明确的页面人格，导致“都像一个页面换了数据”。
4. 科技感主要停留在颜色、毛玻璃和渐变层，尚未形成品牌级视觉语言。
5. `Weekly` 页仍不够像“周报编辑台”，而更像“趋势信息列表页”。
6. `Projects` 页仍不够像“研究工作台”，扫描、比较、下钻的节奏不够强。
7. `Knowledge Base` 页仍不够像“阅读器”，首屏摘要价值与正文编排还不够突出。
8. 旧的页面骨架仍保留较强“通栏区块 + 顺序堆叠”的痕迹，不够颠覆现有可视化框架。
9. 设计系统在需求层没有被完整冻结，后续实现容易再次退回零散样式补丁。
10. `Overview` 首屏模块数量过多，且多数以相似权重的独立白卡出现，导致视觉主次被抹平，用户很难一眼抓住“今天到底先判断什么、然后做什么”。
11. `Weekly` 仍然允许把 `Core Trend` 与 `Weak Signals` 放在同层并排消费，导致最重要的主趋势叙事只能占半屏宽度；同时大卡、小卡和提示卡混排，使页面既拥挤又缺少单一主舞台。

### 1.3 为什么现在必须做整站级改版

`Trend Radar` 的价值不是简单展示分数，而是输出“可信、可解释、可追因、可沉淀”的趋势判断。  
如果外观仍停留在普通后台样式，会直接削弱以下体验：

- 用户对“结果可信度”的第一印象
- 用户对“重点趋势与重点项目”的识别速度
- 用户持续使用和回看的产品意愿
- 这个项目作为一个成熟 AI 趋势产品对外展示时的完成度

因此，本次必须把视觉改造提升为“全站产品换代”，而不是继续做局部皮肤层修补。

## 2. 改版目标

本次 UI / UX 重设计必须同时达成以下目标：

1. 把全站升级为更前沿、更科技感、更有品牌辨识度的产品级界面。
2. 从根本上打破当前“单一页面骨架复用所有视图”的观感，形成真正的页面分工。
3. 让用户在 `10-20` 秒内完成“这次结果能不能信、今天/本周先看什么”的首轮判断。
4. 让 `Weekly`、`Projects`、`KB` 拥有明显不同但同源的体验气质。
5. 让整站具备可持续演进的设计系统，而不是继续依赖局部样式堆叠。
6. 用更强的信息编排与视觉叙事，体现 `Trend Radar` 的“趋势雷达/情报台”身份。
7. 在不破坏 `artifact-first`、状态模型与可审计边界的前提下完成全面外观升级。

## 3. 非目标与边界

### 3.1 非目标

本次明确不做以下事情：

1. 不重算评分、趋势、风险、freshness、verify 语义。
2. 不引入新的业务真相源或新的后端平台。
3. 不把本次需求升级为完整 SPA 架构迁移项目。
4. 不以“炫技特效”替代真实信息效率。
5. 不把界面做成通用 admin template、通用 BI dashboard 或大屏风格。

### 3.2 设计边界

本次改版允许“颠覆外观框架”，但不允许“颠覆业务语义框架”：

- 可以彻底改页面骨架、卡片组织、排版气质、视觉系统、动效节奏。
- 不能改 `Overview / Projects / Weekly / Run Health / KB` 的核心对象语义。
- 可以新增更高级的承载方式。
- 不能让 UI 自己推导出新的项目结论或趋势结论。

## 4. 用户与关键使用场景

### 4.1 用户角色

1. 技术趋势观察者：快速发现今天和本周值得继续追踪的项目与方向。
2. AI 系统工程师：更关注 `agent runtime`、`memory`、`autonomy`、`tooling`、`governance` 等范式变化。
3. 系统维护者：优先查看当前结果是否可靠、是否存在 source 失败、fallback 与可疑状态。
4. 深度研究用户：需要从趋势卡进入项目详情、评分证据、KB 与历史沉淀。
5. 对外展示或评审用户：需要看到一个足够成熟、足够统一、足够前沿的产品界面，而不是实验性质页面。

### 4.2 核心场景

1. 打开 `Overview` 后，在首屏判断这次结果是否值得继续读。
2. 打开 `Projects` 后，像在研究工作台里快速扫描、比较、筛选和下钻项目。
3. 打开 `Weekly` 后，像在读一份编辑化周报，迅速抓住主趋势、弱信号和后续观察点。
4. 打开 `Run Health` 后，像在看一页可操作的系统体检报告，而不是日志堆叠。
5. 打开 `Knowledge Base` 后，像进入一个研究阅读器，而不是一个字段展示页。
6. 在桌面端与移动端都能自然阅读详情，不被壳层和滚动冲突打断。

### 5.1 参考后冻结的设计倾向

结合以上标杆，本项目视觉方向冻结为：

1. 不走传统后台模板路线。
2. 不走单纯“渐变毛玻璃炫酷页”路线。
3. 走“前沿开源开发者产品 + 情报控制台 + 编辑化研究阅读”的混合路线。
4. 更像一个可信、专业、安静但不乏锋芒的工程产品。

## 6. 设计愿景与总原则

### 6.1 设计愿景

`Trend Radar` 应该让用户感觉自己进入的是一个“正在运转的趋势研究工作台”，而不是打开了一页汇总卡片。

### 6.2 设计关键词

1. 科技感
2. 情报控制台
3. 研究工作台
4. 编辑化阅读
5. 深度分层
6. 高密但不压迫
7. 克制动效
8. 品牌化而非模板化

### 6.3 必须坚持的设计原则

1. `Trust First`：可信度、上下文、状态优先于装饰。
2. `Strong Personality per Route`：每个一级视图都必须有自己的页面人格。
3. `One System, Many Surfaces`：五个页面必须是同一套系统，不允许各自为战。
4. `Editorial + Analytical`：既要能读，也要能扫。
5. `Depth, not Noise`：使用深度层次，而不是用过量颜色和过量动画堆刺激。
6. `High-End Open Source`：气质要接近优秀开源产品，而不是企业内网工具。

## 7. 必须打破的旧框架

为了实现“全面整体改变外观”，以下旧模式必须被明确废弃：

1. 不再使用“所有页面共享同一套通栏大卡片顺序堆叠”的主骨架。
2. 不再让每个页面都像“同一页换了标题和数据”。
3. 不再让 `Preference / 设置` 模块长期占据首屏最大面积。
4. 不再依赖“文字 + 文字 + 文字”的线性堆叠承载所有信息。
5. 不再让 `Weekly` 与 `KB` 继续看起来像“列表页”和“字段页”。
6. 不再把雷达、趋势、状态等微型可视化当成装饰角标，而要成为编排的一部分。
7. 不再让整站停留在“漂亮一点的管理后台”这一层级。

## 8. 全站信息架构与壳层要求

### 8.1 全局壳层定位

全局壳层必须更像一个“产品操作系统外壳”，而不是网页导航条。

### 8.2 全局壳层必须包含

1. 品牌标题区：清晰表达 `Trend Radar` 是一个趋势情报产品。
2. 一级导航区：五个视图需要具有明确切换层次。
3. 上下文条：始终显示日期、时间窗、模式、artifact 时间、状态摘要。增加“时间漫游（Time Navigator）”设计，支持流畅的全局历史切片穿梭（如 Yesterday / Last Week），而不仅是静态的文字日期。
4. 快速动作区：保留语言、主题、未来命令入口或快捷跳转空间。
5. 焦点层：详情阅读面板、选中项、保存反馈要形成稳定焦点视觉层。

### 8.3 全局壳层视觉要求

1. Header 更像 floating workspace shell，而不是传统网站头部。
2. Context Strip 必须更紧凑、更像“运行上下文仪表条”。
3. 页面背景要提供稳定的科技氛围层，但不能喧宾夺主。
4. 详情面板必须像“研究侧栏/阅读侧栏”，不是另一整页。

### 8.4 时间漫游引擎 (Time Navigator) 设计规范

考虑到 `Trend Radar` 的核心价值是“时间线上的局势演变”，“时间漫游”不能仅仅是一个原生日期选择器（Date Picker），而必须升级为全局的核心控制组件：

1. **P0：全局位置与折叠态**：嵌入在 Context Strip 中。默认以类似标签/药丸的 UI 简明展示当前所处时间切片、运行状态与 artifact 时间（如 `2026-05-11 · [Data: Stable]`），不能只显示抽象的“latest”。
2. **P0：基础步进交互**：除直接选择日期外，必须提供类似媒体控制器的极简“前一天 (Prev)”、“后一天 (Next)”与“回到最新 (Jump to Latest)”交互。要求结合键盘快捷键（如 `[` 与 `]`），允许用户像翻阅胶片一样无缝步进。
3. **P1：展开式历史预览 (Preview)**：展开后可呈现带有数据热力提示或波动曲线的“微型时间轴面板（Timeline / Heatmap）”。时间轴或日历节点上的状态预告只能来自既有 artifact 或其轻量索引，不得由前端自由推断。
4. **P0：数据平滑过渡动效**：时间维度的切换不应当触发页面的强行刷新（Hard Reload）和刺眼的画面清空。需要配合全局的“柔和数据剥离与注入”转场动效（如内容轻微 blur 或数据的滑入幻影），保证用户能维持“坐在同一个监控台前查阅过去”的心流连贯性。

### 8.5 时间漫游边界约束

为避免时间漫游在实现阶段越界为新的业务解释层，必须冻结以下约束：

1. **步进语义冻结**：`Overview / Projects / Run Health / Knowledge Base` 等单日上下文视图按 `date` 维度步进；`Weekly` 视图按 `anchor` 维度步进。步进时优先跳到“下一个存在可用 artifact 的切片”，而不是机械假定每天或每个时间窗都存在有效结果。
2. **预览数据来源冻结**：时间轴节点上的热力、颜色、波动曲线、状态点和“强趋势日”提示，只允许消费既有 artifact 或由既有 artifact 轻量汇总得到的索引层信息；前端不得根据自由文本、临时规则或页面内容自行推断“强趋势日”“稳定日”“异常日”。
3. **URL 语义冻结**：每次时间漫游都必须落到显式 URL 参数上。单日视图使用 `date=YYYY-MM-DD`，周视图使用 `anchor=YYYY-MM-DD`；`latest` 只能作为快捷入口，解析后必须回写为显式日期或显式 anchor。
4. **上下文继承冻结**：时间漫游后必须尽可能保留当前合法上下文，如 `project`、`trend_key`、`slug`、`source_view`、`theme`、`lang`。若目标切片下该对象不存在，则必须显式降级或清理，不得静默制造错误映射。
5. **历史真实性冻结**：时间漫游只允许消费目标日期或目标时间窗真实存在的 artifact，不得把最新结果伪装为历史结果，也不得在 UI 层自动补造缺失历史。
6. **状态可见性冻结**：若目标切片不存在 artifact、缺少关键审计信息、或结果不可用于判断，时间漫游必须显式展示 `empty / failed / stale / degraded / not-judgeable` 对应语义，不得用平滑动画掩盖状态变化。
7. **性能边界冻结**：时间漫游必须优先采用局部更新与上下文恢复，不应因为单次步进造成整页硬刷新、重复同步读盘或明显卡顿；但任何性能优化都不得改变 `artifact-first` 边界。

## 9. 页面级需求

## 9.1 Overview

### 页面定位

`Overview` 必须像“今日判断首页”或“情报总控面板”。

### 页面目标

1. 第一眼先回答“靠不靠谱”。
2. 第二眼告诉用户“先看什么”。
3. 第三眼告诉用户“为什么”和“下一步是什么”。

### 页面要求

1. 必须采用非对称首屏结构，避免均质卡片墙。
2. `Run Trust Summary` 必须成为视觉主角之一。
3. 首屏默认只允许三个阅读区：`Judgment Canvas`、`Continuation Rail`、`Research Stream`；不得再冻结成五六个彼此并列的卡片位。
4. `Run Trust Summary` 与 `What To Watch First` 必须形成一个连续的 `Judgment Canvas`，而不是两个相互漂浮的独立白卡。
5. `Top Decisions` 在首屏只允许以一个 featured decision 形式出现；完整项目流必须下沉到 `Decision Stream`，不能在首屏堆多张同类卡片。
6. `Weekly Entry` 必须具备高质量入口感，但要收敛到右侧连续行动区里，不得再做成单独抢戏的大卡。
7. `Risks & Next Actions` 必须是精简的后续动作提示，不得扩张为与主判断等权的大块内容。
8. `Source Ribbon` 必须是压缩状态条或微卡阵列，只承担来源健康补充，不得再做成大面积卡片墙。
9. `Preference / Interests` 不再属于 `Overview` 的 P0 内容模块；它应移入全局壳层、设置抽屉或折叠控制区，而不是占据首屏版面。
10. 首屏表面预算必须冻结：除壳层外，默认只允许一个 `Judgment Canvas` 主舞台、一个 `Continuation Rail` 连续行动侧轨和一个 `Research Stream` 研究入口区；属于同一路径的内容必须并入连续表面，不得再拆成多张等权大卡。
11. **Insight 打断机制（Alert Mode）**：当上游 daily artifact、run summary 或等价审计产物显式提供 `editorial_spotlight`（或等价冻结字段）时，支持打破常规的非对称布局，通过放大或提级展现核心警报信息。UI 不得依据自由文本、星数阈值或前端临时规则自行推断“重大范式转移 / S 级项目”。

### 禁止事项

1. 不能把首页做成单纯 KPI 总览页。
2. 不能让首屏同时出现过多等权重模块。
3. 不能把 `Overview` 继续做成“多个白卡平铺”的模块拼盘。
4. 不能让 `Preference`、`Source`、`Weekly Entry` 都以独立大卡身份争抢首屏注意力。
5. 不能让首屏出现大、中、小卡片混排且彼此都试图自成模块的拼贴式排版。

## 9.2 Projects

### 页面定位

`Projects` 必须像“研究工作台”，而不是“项目清单页”。

### 页面目标

1. 支持快速扫描。
2. 支持横向比较。
3. 支持低成本下钻到详情、证据、KB。

### 页面要求

1. 列表区与详情区必须形成明显的双栏工作台关系。
2. 列表项需要具备强识别的主标题、副信息、状态、微型趋势线和快速标签。
3. 筛选器要像研究过滤器，而不是传统表单。
4. 详情区要强调“解释为什么值得看”，而不是把字段重新排版一遍。
5. 页面整体要体现“速度感”和“工程分析感”。

### 禁止事项

1. 不能退化为普通表格后台。
2. 不能让详情阅读打断列表扫描节奏。

## 9.3 Weekly

### 页面定位

`Weekly` 必须像“周趋势编辑台”或“编辑化周报页”。

### 页面目标

1. 快速讲清楚这周真正的主趋势。
2. 区分核心趋势与弱信号。
3. 把证据关系讲清楚，而不是仅罗列项目。

### 页面要求

1. 页面首屏必须具备“周报封面”气质。
2. `Overall Weekly Judgment` 要有主叙事位置。
3. 核心趋势卡必须构成全宽 `Core Trend Stage`，更像 editorial card，而不是 article list，也不得与 `Weak Signals` 平分画面。
4. “分数与证据”区域必须成为本页标志性模块，且位于 `Core Trend Stage` 之前。
5. 大六边形雷达图与六张证据卡必须建立一一映射，且映射关系一眼可读；雷达六轴值必须由上游 weekly 产物为每个 core trend 显式提供，不得由 UI 临时聚合 supporting projects 后自行计算。
6. 若某一轴证据不足，上游必须显式返回 `insufficient_evidence` 或等价审计状态，UI 以缺证形式表达，不得补算成看似完整的雷达面。
7. 页面必须显式给出“下周是否值得继续跟踪 / 后续观察点”，该信息必须绑定到 weekly artifact 的冻结字段，而不是由 UI 临场总结。
8. 若当周没有稳定趋势，也要以高级、诚实的方式表达“暂无稳定主趋势”。
9. `Weak Signals` 必须降为后置次级区，只能出现在 `Core Trend Stage` 之后；当存在稳定核心趋势时，不得与核心趋势卡共享同一横向带宽或同一等权分栏。
10. **强化趋势变化向量（Delta）**：着重通过视觉符号（如上涨/下跌/持平指针、鲜明的相对变化动效）展现“变化量”。但只有当上游 weekly 产物提供可审计的 delta 摘要或结构化变化输入时，UI 才能展示方向性符号；否则必须显式展示 `delta unavailable / 样本不足`。

### 禁止事项

1. 不能继续像普通列表页。
2. 不能把 `Weekly` 当成 daily 的附属结果。
3. 不能在存在稳定核心趋势时，让 `Weak Signals` 与 `Core Trend` 以左右等权双栏共用同一主故事带。

## 9.4 Run Health

### 页面定位

`Run Health` 必须像“系统体检与审计页”。

### 页面目标

1. 快速定位问题。
2. 快速识别是否可继续信任当前结果。
3. 快速理解建议动作。

### 页面要求

1. 页面语气可以更冷静、更结构化，但仍需保持同一品牌语言。
2. `Verify Result Summary`、`Run Summary / Trust Gate`、`Source Status Table`、`Audit Table` 之间要有清楚的诊断流程感。
3. 失败、空态、降级、不可判断必须有明确区分。
4. 需要具备“运维面板 + 审计面板”的结合气质。

### 禁止事项

1. 不能做成原始日志页。
2. 不能通过颜色 alone 表达严重程度。

## 9.5 Knowledge Base

### 页面定位

`Knowledge Base` 必须像“研究阅读器”。

### 页面目标

1. 首屏先给结论摘要与阅读价值。
2. 再允许继续进入机器区与人工区。
3. 保证长期阅读体验。

### 页面要求

1. KB 首屏必须有 `Executive Summary` 风格摘要。
2. 机器区与人工区必须明显区分，但都要可读。
3. `Linked Context` 必须让用户知道这张卡和当前 daily / weekly / project 的关系。
4. 需要支持更宽的 reader mode 或更强的正文承载感。
5. 页面要更接近“专业文档阅读体验”，而不是管理台侧栏。

### 禁止事项

1. 不能继续是字段堆叠页。
2. 不能让 KB 阅读区看起来像附属玩具面板。

## 10. 视觉系统需求

### 10.1 主题系统

1. 必须有高质量 Light / Dark 双主题。
2. 浅色主题不能像传统企业后台。
3. 深色主题不能退化为纯黑纯白高对比页面。
4. 主题系统要支持未来继续扩展品牌变体。

### 10.2 字体系统

1. 标题字体必须更有个性与科技感。
2. 正文字体必须适合中英文混排。
3. 数字、状态、指标必须使用等宽数字策略。
4. 字体体系要体现“研究产品”而不是“文档工具默认字体栈”。

### 10.3 色彩系统

1. 主色要体现科技与可信，不走俗套紫色模板路线。
2. 辅助色用于趋势、成功、告警、风险，但必须克制。
3. 状态色需要统一语义，并配合文案标签使用。
4. 整站需要形成稳定的品牌识别，而不是每页换一套强调色。

### 10.4 版式系统

1. 需要明确的大中小屏栅格体系。
2. 需要稳定的页面最大宽度与正文阅读宽度。
3. 需要更高级的非对称布局与层级留白。
4. 需要统一的圆角、边框、阴影、模糊与层次规范。

### 10.5 图形与图表语言

1. 微型图形必须简洁、信息化、可嵌入版面。
2. 雷达图、sparkline、heat bar 要形成统一视觉语言。
3. 图形是“信息编排的一部分”，不是附着装饰。
4. 不引入与产品气质冲突的大面积商业 BI 图表风格。

### 10.6 AI 透传与事实边界 (AI Transparency)

1. 必须在视觉上区分“客观抓取数据”（如硬指标事实）与“AI / 规则生成的判断说明”。
2. 本次不引入新的抽象“AI confidence score”；透明度必须直接建立在现有审计字段之上，而不是再发明一套前端置信度体系。
3. 页面级透明度必须使用 `enhancement_status`（如 `rules-only / agent-partial / agent-full`）说明本次结果属于完整增强、部分增强还是规则保底。
4. 字段级透明度必须使用 `enhancement_source`、`risk_review_source` 等来源标签，区分具体文案来自 Agent 还是模板 fallback。
5. 审计级透明度必须保留 `rejected_outputs[]` 或等价审计入口，使用户能追溯“为什么该处没有采用 Agent 输出”。

### 10.7 降级与边界状态美学 (Edge States)

1. **空状态（Empty）**：抛弃简陋的“No Data”，应设计符合防备台冷寂感的静态美 UI（例如“今日无重大异动”的高级留白）。
2. **错状态（Error）**：API 宕机等异常需避免原生警告红，应采用克制优雅的视觉降级。
3. **等待状态（Loading）**：用类似脉冲波、扫描线的“微型仪器态”动效，替代传统的转圈 Spinner。

## 11. 交互与动效需求

### 11.1 总原则

动效必须“稳、轻、准”，服务理解，不制造噪音。

### 11.2 必做要求

1. 路由切换要有平滑但低干扰的状态反馈。
2. 详情展开、保存反馈、悬停提升都需要明确而克制的动效。
3. Skeleton 与 loading 状态要比当前更产品化。
4. 所有可交互元素都必须有清晰 hover 与 `focus-visible` 状态。
5. 键盘工作流要被认真支持，尤其在 `Projects` 与 `Preference` 相关区域。

### 11.3 可选增强

以下增强项建议纳入设计储备，但优先级低于主骨架与主页面：

1. 全局快捷切换或命令面板入口
2. 日期/时间窗快速跳转器
3. 更丰富的列表密度切换
4. 时间漫游增强态：展开式微型时间轴与热力预览
5. Overview 的传播友好能力：复制 Markdown、导出摘要卡片
6. Weekly 的对外分发增强：总览长图或分享卡
7. Run Health 的高效排障增强：复制诊断日志、建议修复路径展示
8. Knowledge Base 的实体穿透增强：内联 Hover Card / 双向链接阅读体验

## 12. 响应式与可访问性要求

### 12.1 响应式

1. 桌面端采用绝对的 **Desktop First** 原则，优先最大化工作台空间。
2. 移动端定位为信息的纯消费与监控延伸，**仅提供优雅的只读降级视图**。移动端详情面板进入全屏阅读模式，不挤压工作台复合逻辑。
3. 平板端要平滑收敛为单列或弱双列。
4. 小屏下关闭和返回路径必须始终清晰可见。

### 12.2 可访问性

1. 不能只靠颜色表达状态。
2. 图形不能成为唯一信息来源。
3. 长文阅读对比度必须足够。
4. 键盘导航路径必须稳定。
5. 中英文混排、长 repo 名、长摘要必须避免破版。

## 13. 性能与工程约束

虽然这是视觉重设计需求，但必须同时冻结以下非功能约束：

1. 视觉升级不能显著恶化一级视图切换性能。
2. 高成本微型可视化必须可延后渲染或按需渲染。
3. 详情切换应优先局部更新，不应频繁整页重建。
4. 任何视觉设计都不能逼迫 UI 越界重算业务逻辑。
5. 组件化、token 化、结构解耦必须成为后续实现方向。
6. 若要支撑 Overview Alert Mode、Weekly 六轴雷达或 Delta 展示，上游必须输出对应的结构化 payload（如 `editorial_spotlight`、per-trend evidence matrix、delta summary）；UI 不得为了满足视觉效果自行补算这些字段。

## 14. 交付优先级

### P0：必须落地

1. 全局壳层换代
2. 五个一级视图统一改版
3. `Overview / Projects / Weekly / Run Health / KB` 页面人格重建
4. `Weekly` 的大六边形雷达 + 证据映射重构
5. `Knowledge Base` 阅读器化
6. 主题系统、版式系统、字体系统、状态系统冻结
7. 桌面与移动端详情承载重构
8. Time Navigator 基础态：显式上下文、`Prev / Next / Latest`、URL 回写与键盘步进

### P1：强烈建议落地

1. 微型趋势可视化体系
2. 更高级的首屏叙事布局
3. 键盘优先交互增强
4. 全局命令面板（Command Palette / CMD+K 快捷跳转核心能力）
5. 更产品化且符合上下文边界的 loading / skeleton / edge states 系统
6. 时间漫游增强态：展开式微型时间轴与热力预览

### P2：后续储备

1. 视图密度自定义
2. 品牌主题扩展
3. Overview 的传播友好能力：复制 Markdown、导出摘要卡片
4. Weekly 的对外分发增强：总览长图或分享卡
5. Run Health 的高效排障增强：复制诊断日志、建议修复路径展示
6. Knowledge Base 的实体穿透增强：内联 Hover Card / 双向链接阅读体验
7. Projects 的多项目平行对比视图（Matrix Compare）

## 15. 验收标准

### 15.1 视觉换代验收

1. 用户一眼就能感知“这不是旧版页面修补，而是一套新的产品外观”。
2. 整站具备明显科技感、产品完成度与品牌感。
3. 五个一级视图统一但不雷同。

### 15.2 体验效率验收

1. 用户在 `10-20` 秒内能完成可信度与重点识别。
2. 用户进入 `Projects` 后能更快完成扫描与下钻。
3. 用户进入 `Weekly` 后能一眼理解主趋势与证据关系。
4. 用户进入 `KB` 后首屏就能读到高价值摘要，而不是先看到结构化字段。
5. “10-20 秒完成首轮判断”必须通过固定 fixture + 任务脚本或等价计时方式验证，而不能只用结构检查代替效率验证。

### 15.3 结构升级验收

1. 页面不再依赖旧的通栏堆叠框架。
2. `Preference` 区不再成为首屏主视觉负担。
3. `Weekly` 评分证据模块不再沿用“小雷达 + 散列表达”的旧方式。
4. `Weekly` 在存在稳定核心趋势时，核心趋势卡必须占据全宽叙事舞台，`Weak Signals` 不得再与其等权并排。
5. 详情承载区在桌面端与移动端都具备明确、顺畅、可持续阅读的结构。

### 15.4 工程可持续验收

1. 设计 token 体系足够完整，可支撑后续继续演进。
2. 样式与组件边界清晰，不再只能靠零散模板拼接维护。
3. 不破坏 `artifact-first`、状态模型与跨视图一致性。

## 16. 需求冻结结论

`READY FOR DESIGN AND IMPLEMENTATION`

本次冻结的不是“页面修成什么颜色”，而是以下产品级结论：

1. 必须做整站级视觉换代，而不是局部美化。
2. 必须参考优秀开源开发者产品的前沿前端设计，但要形成 `Trend Radar` 自己的情报工作台语言。
3. 必须打破现有单一页面骨架，让五个一级视图真正拥有不同的人格与节奏。
4. 必须把 `Weekly` 和 `Knowledge Base` 变成最能体现产品高级感的页面。
5. 必须把设计系统、动效、响应式、可访问性和性能约束一起冻结，避免后续只做“漂亮截图式 UI”。

## 17. 下一步建议

在本需求文档之后，建议按以下顺序继续推进：

1. 基于本 PRD 输出视觉方向板与页面级 moodboard。
2. 产出全局 design token、字体、色彩与组件规范。
3. 先做 `Overview / Projects / Weekly` 三页高保真稿。
4. 再补 `Run Health / Knowledge Base` 与移动端版本。
5. 最后进入实现与真实浏览器视觉回归。
