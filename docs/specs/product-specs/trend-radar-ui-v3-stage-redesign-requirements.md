## 一句话冻结

`UI-V3` 不是继续给现有玻璃卡片“换皮”，而是把 `Trend Radar` 从“可读的卡片控制台”升级为“有主舞台、有路线节奏、有研究氛围的趋势情报工作台”。

本次会明确借鉴 `pvp.qq.com` 首页的前端语法，但只借鉴结构和气质，不复制任何游戏资产、IP 元素、角色视觉或装饰语汇。

## 1. 背景与问题定义

### 1.1 当前阶段判断

现有 `UI-V2.5` 已经解决了“有没有 Web 控制台”的问题，也具备了以下基础：

- 五个一级视图已经成型：`Overview`、`Projects`、`Weekly`、`Run Health`、`Knowledge Base`
- `artifact-first`、时间导航、详情侧栏、主题切换、双语切换已具备基础实现
- 页面信息量已经足够支撑研究、巡检和阅读任务

当前最大的缺口不在功能，而在前端承载方式：

- 结构上仍然是卡片堆叠
- 情绪上仍然像“升级过的后台”
- 编排上仍然缺少“先看什么，再深入什么”的舞台引导

### 1.2 当前 UI 的核心问题

1. 首屏缺少单一主舞台，用户打开后第一眼无法迅速建立判断优先级。
2. 各路由虽然内容不同，但视觉框架过于同质，仍然像“一个页面换了数据”。
3. 视觉重心仍在卡片容器本身，而不是内容层级、叙事节奏和阅读路径。
4. 科技感主要来自玻璃、渐变、圆角和阴影，缺少更明确的“情报控制台”视觉语言。
5. `Overview` 首屏模块过多且权重接近，仍然存在“什么都重要，于是什么都不重要”的问题。
6. `Projects` 更像卡片列表，不像高效率研究工作台；扫描、对比、下钻三段节奏不够强。
7. `Weekly` 已经有内容结构，但仍然不够像“周趋势编辑台”，核心趋势与弱信号的舞台主次还不够绝对。
8. `Run Health` 信息可用但不够像诊断面板，可靠性判断链条没有被做成明显的操作路径。
9. `Knowledge Base` 仍偏字段阅读和侧栏消费，不够像研究型 dossier reader。
10. 视觉语义仍偏“漂浮内容块”，缺少边框、框景、导视、分段锚点与层层推进的前端节奏。

### 1.3 为什么以 `pvp.qq.com` 为研究输入

基于 `2026-05-15` 对 [pvp.qq.com](https://pvp.qq.com/) 首页的研究，值得借鉴的不是“游戏官网视觉”，而是以下前端表达能力：

1. 首屏先给主舞台，再给次级入口，不先堆内容块。
2. 顶部导航像浮在舞台上的控制层，而不是独立白卡。
3. 大模块之间存在明显“段落感”和“场景切换感”，不是均质列表。
4. 视觉重心依赖背景氛围、边框框景、强调分段和焦点区域，而不依赖纯色卡片。
5. 页面有强主次和清晰导视，用户总能知道“当前重点是什么”。

这正是 `Trend Radar` 当前缺少的前端能力。

### 1.4 当前偏航复盘（2026-05-20）

本轮 UI-V3 落地之所以会让人感觉“完全背离初衷”，问题不在颜色、字体或质感细节，而在于实现阶段把“页面语法升级”错误地收窄成了“壳层重命名 + panel/card 样式重写 + surface role 断言通过”。

具体偏航点如下：

1. 首屏构图没有真正切换到现代首页语法，仍然是“大 header 容器 + 路由入口拼盘 + 指标条幅”的后台式排布。
2. 执行时虽然引入了 `hero / stage / rail / dock` 命名，但这些语义没有转化成真实的 viewport 主次关系，导致 DOM 结构看似 V3，实际首屏阅读路径仍是 V2.5。
3. 验收与 visual regression 过度依赖类名、surface id、顺序和局部 computed style，缺少“第一屏截图是否真的像现代网站首页”的构图验收。
4. 因此最终页面更像“旧控制台换了更精致的皮肤”，而不是“小红书 Explore / 王者荣耀官网这类先建立主舞台、再组织次入口、再推进内容段落”的现代网页。

本次修订的目标，是把“现代网页布局优先于面板质感优化”重新冻结为不可误解的产品要求。

### 1.5 参考图澄清：目标不是泛泛“现代”，而是现代控制中台语法

用户补充的参考图进一步澄清了 `UI-V3` 想要的不是抽象意义上的“现代感”，而是一种更具体的现代控制中台 / operations console 页面语法。该语法至少包含以下可观察特征：

1. 壳层是“薄”的：顶部控制条、全局搜索、状态位、用户区、侧边导航都属于操作壳层，而不是巨型首页 hero。
2. 主内容是“厚”的：真正占据首屏主导位的是业务主舞台，例如 KPI 带、主图谱、地图、主列表、主分析面板，而不是路由入口卡片。
3. 导航是“基础设施”，不是“内容模块”：路由切换应更像左侧导航或顶部控制条，而不是四张并列的入口卡。
4. 信息组织是“先总览，再主分析，再明细工作区”的控制台节奏，而不是“先看壳层说明，再看一些次级入口，再进入内容”。
5. 页面必须有明显的工作台密度、框架边界和模块纪律，避免大面积空白 hero、漂浮玻璃卡和演示型首页构图。

这意味着 `Trend Radar` 的目标更接近“趋势情报控制台”，而不是营销首页，也不是仅靠氛围和 Hero 文案支撑的内容首页。

## 2. 研究结论与转译原则

### 2.1 从参考站点提炼出的可复用模式

1. `Hero First`：首屏先给主叙事舞台，再给通往其他内容的入口。
2. `Floating Shell`：导航、状态和工具层浮在内容之上，像操作台壳层，而不是单独卡片。
3. `Framed Sections`：每一段内容都有明确框景、边界和标题区，不靠空白堆出层级。
4. `Atmosphere Layer`：背景不是纯色，而是稳定、可控、低噪音的氛围层。
5. `Segmented Story Flow`：模块之间是“段落推进”，不是“等权列表”。
6. `High Contrast Focus`：重点区域必须在尺寸、对比和节奏上显著高于次要信息。

### 2.2 本项目的转译原则

1. 借鉴结构，不借鉴游戏题材。
2. 借鉴舞台感，不复制角色海报、金边古风纹理或宣传页动效。
3. 借鉴首屏焦点，不引入与业务无关的大面积视觉噪音。
4. 借鉴分段导视，不牺牲 `artifact-first` 和可审计语义。
5. 借鉴沉浸式壳层，但必须保留开发者产品应有的克制、可信和可读性。
6. 整站主气质必须以 `Intelligence Terminal / 情报终端` 为主，而不是平均分配给多种页面人格。
7. `Weekly` 可以更明显吸收 `Editorial Desk / 编辑台` 语气，`Knowledge Base` 可以更明显吸收 `Research Reader / 研究读板` 语气，但二者都必须从属于同一套情报工作台语言。
8. UI-V3 不只定义“长什么样”，还必须定义“信息先后如何登场、焦点如何迁移、用户如何沿着舞台继续下钻”。

### 2.3 明确禁止项

1. 不允许直接模仿 `pvp.qq.com` 的素材、图像、角色、字体商标或装饰符号。
2. 不允许把产品做成游戏官网、活动页或营销页面。
3. 不允许只把当前卡片改成“深色 + 金边”后就视为 `UI-V3`。
4. 不允许为了舞台感牺牲信息密度、可读性、键盘操作和状态透明度。

## 3. UI-V3 总目标

1. 让整站从“卡片集合”升级为“主舞台 + 侧轨 + 细节读板”的工作台式体验。
2. 让用户在 `10-15` 秒内完成两件事：
   - 当前结果能不能信
   - 今天或本周应该先看什么
3. 让五个一级视图拥有不同的人格，但共享同一套设计系统。
4. 让 `Overview`、`Projects`、`Weekly` 都摆脱均质卡片堆叠。
5. 让 `Run Health` 和 `Knowledge Base` 从“附属页”升级为有独立气质的专业界面。
6. 让视觉语言更锐利、更有框架感、更像“情报控制台”而不是“玻璃后台”。
7. 在不改业务真相边界的前提下，为后续实现 `UI-V3` 提供明确需求冻结。

## 4. 非目标与边界

### 4.1 非目标

1. 不重算评分、趋势、风险、freshness、verify 或 weekly 判定逻辑。
2. 不引入新的业务真相源、媒体资源系统或 CMS。
3. 不在本阶段做完整前端架构迁移或 SPA 重写。
4. 不用炫技特效代替信息组织。
5. 不把 `Trend Radar` 做成游戏风、宣传风或通用 dashboard 模板。

### 4.2 设计边界

1. 可以重做页面骨架、区域布局、视觉层次、动效和设计 token。
2. 不能改变五个一级视图的对象语义和 artifact 消费边界。
3. 可以新增更适合承载内容的容器、导视和过渡层。
4. 不能让 UI 自己推导新的事实、趋势或结论。
5. 不能把移动端目标定义成“完整桌面复制”；移动端允许优雅降级为偏阅读视图。
6. 可以重构前端 SSR render 层、组件层与样式分层，以建立更稳定的 `stage / rail / dock / reader` 语义承载。
7. 不能让 render 层反向侵入 read layer、context layer 或 truth layer；视觉重构不应改写 artifact-first 的分层边界。
8. 可以新增用于焦点迁移、证据锚点、阅读密度切换的前端承载模式，但这些模式不得引入第二套排序真相或前端聚合真相。

## 5. 核心设计愿景

### 5.1 产品气质

`Trend Radar UI-V3` 应该让用户感觉自己进入的是一个“趋势情报工作台”：

- 有主舞台
- 有判断入口
- 有阅读深度
- 有框架感
- 有操作感
- 有研究氛围

### 5.2 设计关键词

1. 舞台感
2. 情报终端
3. 编辑台
4. 研究读板
5. 锋利框架
6. 分段推进
7. 强主次
8. 锚点导视
9. 克制动效
10. 内容优先

### 5.3 必须坚持的原则

1. `Stage Before Cards`：先有舞台，再有卡片。
2. `Trust Before Exploration`：先判断可信度，再进入深层内容。
3. `One Route, One Persona`：每个路由都有明确的人格和节奏。
4. `Frame and Rail, Not Card Wall`：优先使用框景、导轨和条带，而不是满屏同权卡片。
5. `Editorial + Analytical`：既能讲清楚，也能高效率扫描。
6. `Signal, Not Spectacle`：视觉服务判断，不服务表演。
7. `Viewport Composition Before Surface Polish`：先解决首屏构图、段落推进和主次节奏，再讨论玻璃、渐变、阴影和局部组件皮肤。

### 5.4 气质主轴与路由人格

1. 全局主气质必须明确偏向 `Intelligence Terminal / 情报终端`，也即更像可信的研究操作台，而不是营销页、设计秀或通用 SaaS 后台。
2. `Overview` 必须是情报终端里的 `decision gate / 判断入口`，负责先建立“能不能信、先看什么”的一阶判断。
3. `Projects` 必须是情报终端里的 `scan bench / 扫描台`，负责高速横向比较与局部下钻。
4. `Weekly` 必须在共享同一套全局语言的前提下，成为更明显的 `Editorial Desk / 编辑台`，强调封面、编排、主叙事和证据骨架。
5. `Run Health` 必须是情报终端里的 `diagnostic audit stage / 诊断审计台`，语气更冷静、更收束、更偏审计面板。
6. `Knowledge Base` 必须在共享同一套全局语言的前提下，成为更明显的 `Research Reader / 研究读板`，强调阅读宽度、脚注、分层正文和长时消费。
7. 不允许五个一级路由只是“同一套容器换数据”；必须做到同系统、不同人格、不同阅读节奏。

## 6. 全局壳层需求

### 6.1 全局结构

全局页面必须稳定分为三层：

1. `Shell Hero`：品牌、当前视图人格、一级导航、轻量控制器。
2. `Context Instrument Bar`：时间、artifact 时间、状态、模式、导航步进。
3. `Content Stage`：按路由切换的主舞台和次级内容区。
4. 若采用侧边导航或压缩顶部控制条来实现更强的“控制中台”语法，视为符合方向；但不得再把壳层做成首屏最大内容容器。

### 6.2 Header / Shell 要求

1. Header 必须更像“悬浮操作台壳层”，不能再像一张大卡片。
2. 一级导航必须有更强的段落感和焦点态，当前路由要明显占据主导位。
3. 当前路由的人格说明必须在首屏可见，而不是只显示标题。
4. 主题切换、语言切换、快捷动作必须收束在壳层工具区，不与主内容争抢舞台。
5. `Interest Profile` 不得长期占据首屏最大视觉权重，应下放为辅助控制区或可折叠工具区。
6. 首屏不得退化成“当前工作面说明 + 四个等权路由入口卡片 + 下方再接 context bar”的 dashboard 导航拼盘；如果用户第一眼仍先看到入口卡而不是主舞台，判定为布局失败。
7. `Shell Hero` 这个命名指的是壳层优先级，不代表它应该成为视觉上最大的 hero。真正的首屏大舞台必须属于 route-level 内容，而不是壳层本身。

### 6.3 背景氛围层要求

1. 页面背景必须提供稳定的沉浸式氛围层，而不是纯色或单一玻璃底。
2. 背景可以包含径向光斑、框线、扫描纹理、微弱格栅或层叠雾面。
3. 背景不得影响正文对比度，不得让阅读依赖模糊滤镜。
4. Light / Dark 主题都必须有明确氛围，不允许浅色主题退化为普通企业后台。

### 6.4 时间导航要求

1. Time Navigator 必须继续保留 `Prev / Next / Latest` 的快速步进能力。
2. 时间导航的视觉形态应更像“仪表条 + 段位器”，而不是普通 pills 和按钮。
3. 当前切片、切片位置、是否 stale 必须在一眼内可理解。
4. Preview 展开区域应更像时间切片盘或候选带，而不是再加一层普通卡片网格。

### 6.5 详情承载要求

1. 详情区必须被明确塑造成 `reader dock / research dock`。
2. 桌面端详情区应像固定停靠的阅读面板，而不是页面上又一列普通内容。
3. 移动端详情区允许进入全屏阅读模式，但返回路径必须始终清晰。

### 6.6 页面语法 / Surface Role Contract

1. `UI-V3` 必须建立稳定的页面承载角色：`Hero`、`Stage`、`Rail`、`Strip`、`Dock`、`Reader`、`Audit`。
2. `Hero` 只允许承载当前视图在首屏最重要的单一判断或主叙事，不得同时并列多个同权 hero。
3. `Stage` 承载主判断、主趋势、主扫描流或主诊断链条，是当前视图的第一消费对象。
4. `Rail` 承载次级上下文、后续动作、总结性 watchpoints 或辅助入口，不得长期在视觉权重上压过 `Stage`。
5. `Strip` 承载时间、来源状态、切片预览、过滤标签或轻量信号，不得膨胀成新的主内容区。
6. `Dock` 承载可选中的详情对象，并保持独立滚动、可关闭、可恢复的焦点关系。
7. `Reader` 承载长文本、研究正文、机器/人工分层和脚注语义，必须有稳定阅读宽度，不得退化成字段块平铺。
8. `Audit` 承载 rejected outputs、verify、fallback、degraded 等可信度解释，不得伪装成普通内容卡。
9. 同一业务对象不得在 `Hero`、`Rail`、`Dock` 中以同等视觉权重重复出现，避免用户误读当前焦点。

### 6.7 承载层架构要求

1. 前端承载层必须能够稳定表达 `Page Frame -> Route Stage -> Surface Role -> Detail Dock` 的层级，而不是依赖一层通用 `panel/card` 再用 CSS 补丁拉开差异。
2. 允许为 `Overview / Projects / Weekly / Run Health / Knowledge Base` 拆出更明确的 route frame、stage section 和 dock section，但不要求迁移为完整 SPA。
3. 组件与样式系统应优先承载“语义角色”，再承载“视觉皮肤”；换言之，应先区分 `hero / rail / dock / reader`，再区分颜色、主题或阴影。
4. 若后续实现仍以 SSR 为主，则 SSR 输出结构本身必须体现页面语法，而不是把页面人格全部压到样式表中。

## 7. 路由级需求

### 7.1 Overview

#### 页面定位

`Overview` 必须是“今日判断主舞台 / 情报终端入口”，不是首页仪表盘。

#### 页面要求

1. 首屏必须出现一个明显主舞台，用于承载 `trust gate + lead decision + next action`。
2. 首屏一级模块数量必须收敛，不得再出现多个等权大卡平铺。
3. 今日最值得看的项目必须以 `spotlight` 方式呈现，不得退化为列表里的第一张卡。
4. `Run Health` 入口和 `Weekly` 入口应作为侧轨或后续动作出现，而不是与主判断同权。
5. 次级项目应组织成连续扫描带、研究条目或紧凑阵列，而不是一组等权浮卡。
6. 风险与建议动作应聚合成短促、可执行的延续区，不应打断主舞台阅读。
7. 页面必须明确体现 `trust gate -> lead decision -> continuation` 的单向阅读路径，避免用户在多个一级入口间来回比较主次。
8. `Source Health`、`Interest Profile` 等辅助信息如需出现，必须降级为 `strip / tool zone`，不得重新长成首屏主卡。

#### 禁止项

1. 不允许首屏再次回到“六七张大卡堆叠”。
2. 不允许 `Preference`、`Weekly Entry`、`Source Health` 等辅助模块与主判断争抢视觉中心。

### 7.2 Projects

#### 页面定位

`Projects` 必须是“研究扫描工作台 / 情报终端扫描台”，不是项目卡片列表。

#### 页面要求

1. 过滤区必须像研究滤镜台，而不是多个普通 chips 容器。
2. 项目行必须更像 `scan row / dossier row`，强调“左侧 thesis，右侧 score cluster”。
3. 页面应突出扫描速度和横向对比，而非单卡独立观看。
4. 当前选中项目必须形成明显焦点态，并与详情阅读 dock 建立紧密关系。
5. 详情区的重点必须是“为什么值得看、证据是什么、风险和下一步是什么”。
6. 列表主行必须在不打开详情时，也能被快速扫读出 `thesis / score cluster / top evidence / primary risk / next action cue`。
7. 列表与详情之间必须存在稳定的焦点联动和证据锚点，而不是“左边一张卡，右边另一个孤立详情页”。

#### 禁止项

1. 不允许退化成纯表格后台。
2. 不允许保留当前这类“每个项目都是一张相似卡片”的主形态。

### 7.3 Weekly

#### 页面定位

`Weekly` 必须是“周趋势编辑台 / 研究封面页”，是 `UI-V3` 承接 `Editorial Desk` 气质的标志性页面。

#### 页面要求

1. 首屏必须具备明确的 `weekly cover stage`，让用户先看到本周主叙事。
2. `Evidence Matrix` 必须是独立标志模块，且在核心趋势正文之前或并列首屏出现。
3. `Core Trend` 必须拥有全宽或近全宽舞台，不得与 `Weak Signals` 等权并列。
4. `Weak Signals` 必须降为后置次级区，只在核心趋势之后消费。
5. `Watchpoints` 必须收束为清晰的后续观察带。
6. 只有当上游 artifact 明确提供结构化 delta 时，前端才能展示方向型变化符号。
7. 页面必须明显比 `Overview / Projects / Run Health` 更具编排感和封面感，但仍属于同一套情报终端系统，不得漂移成宣传页。
8. `Evidence Matrix`、六轴视图和证据 tile 之间必须存在稳定的编号、轴位或锚点映射关系，做到一眼可对照，而不是只靠空间邻近暗示相关性。

#### 禁止项

1. 不允许再像信息列表页。
2. 不允许在有稳定核心趋势时，把 `Weak Signals` 放到同一主舞台分栏里。
3. 不允许把最重要的核心趋势正文压缩成半屏或次级卡，导致“编辑台”人格只停留在装饰层。

### 7.4 Run Health

#### 页面定位

`Run Health` 必须是“诊断与审计面板 / 情报终端诊断台”，而不是状态列表页。

#### 页面要求

1. 页面必须让用户按固定顺序完成判断：
   - 结果总体能不能信
   - 哪些 source 正常/失败
   - verify 是否通过
   - 下一步该做什么
2. 诊断链条必须以舞台式分段呈现，而不是多个内容块并列。
3. 表格仍可保留，但外层必须有更强的诊断框景与状态引导。
4. `failed / degraded / stale / empty` 必须在文字、颜色、图标或标签上可区分，不能只靠颜色。
5. 页面必须更像 `audit stage` 而不是普通数据页，至少在段落顺序、状态警示和建议动作层次上体现“先诊断再行动”。

#### 禁止项

1. 不允许变成原始日志页。
2. 不允许和其他路由共享同一套“玻璃卡堆叠”骨架。

### 7.5 Knowledge Base

#### 页面定位

`Knowledge Base` 必须是“研究 dossier 阅读器 / 研究读板”，不是字段详情页。

#### 页面要求

1. 首屏必须先给 `executive summary`，再进入机器区和人工区。
2. 页面应更像 reader，而不是管理后台的 detail panel。
3. `Machine Notes` 与 `Human Notes` 必须在视觉上明显分层。
4. 关联上下文必须被收束到阅读脚注、meta rail 或 reader footer，不应打断正文。
5. 如卡片较轻量，应明确标识为 lightweight brief，而不是伪装成完整研究条目。
6. 页面正文必须优先建立 `summary -> body -> linked context` 的自上而下阅读顺序，而不是横向消费多个字段面板。
7. `Knowledge Base` 的宽度、字号、行长、脚注和 meta rail 必须优先服务长时间阅读，体现 `Research Reader` 而非后台明细页。

#### 禁止项

1. 不允许继续以字段块堆叠为主要阅读形态。
2. 不允许让 KB 看起来像“项目详情附属页”。

## 8. 设计系统需求

### 8.1 色彩系统

1. 主题方向应采用“深底 + 金属高光 + 信号色点亮”的策略，而不是紫色模板化科技风。
2. Dark 主题建议偏 `ink / graphite / brass / cyan signal`。
3. Light 主题建议偏 `stone / parchment / charcoal / brass accent`。
4. 风险、警告、成功、信息状态都必须有统一语义色，不得每页自行发挥。
5. Light 主题不得只是把 Dark 主题“翻白”；必须保留纸面、台面、框景和信号层的材质感，避免退化为普通企业后台。

### 8.2 字体系统

1. 标题字体必须更有个性和舞台张力，不能再只是普通无衬线默认组合。
2. 正文字体必须适合中英混排和长时间阅读。
3. 数字、时间、指标和状态必须坚持等宽数字策略。
4. 字体气质应指向“研究产品 / 情报控制台”，而非“活动页字体秀”。

### 8.3 组件系统

`UI-V3` 至少要冻结以下核心承载组件：

1. `Stage Hero`
2. `Framed Panel`
3. `Context Instrument Strip`
4. `Route Tabs / Intent Tabs`
5. `Scan Row`
6. `Evidence Tile`
7. `Reader Dock`
8. `Status Band`
9. `Meta Rail`
10. `Signal Chip`
11. `Stage Frame`
12. `Audit Panel`
13. `Reader Frame`
14. `Evidence Anchor`

### 8.4 页面语法与微视觉语言

1. 页面高级感必须更多来自框景、刻度、编号、状态条、分段锚点、mono 指标和节制高光，而不是来自玻璃、模糊和大面积渐变。
2. 页面中的编号、轴位、脚注、引用、source 状态和趋势证据必须尽量使用一致的微视觉锚点语言。
3. `Weekly` 的六轴矩阵、`Projects` 的扫描行、`Run Health` 的诊断步骤、`KB` 的脚注区应共享同一套微型导视逻辑，而不是各页自创一套图形符号。
4. 任何 decorative layer 都必须为阅读层级服务；若视觉细节不能提升判断速度，则不应进入 P0。

### 8.5 版式系统

1. 页面必须更多使用非对称双栏、主舞台 + 次轨布局。
2. 一级内容区应限制真正等权并列模块的数量。
3. 首屏必须优先建立主次，不得把所有信息平均分配。
4. 圆角、边框、发光、模糊、描边和分隔线必须 token 化，禁止继续散落堆叠。
5. 不同路由的人格差异应优先通过 `surface role`、阅读节奏、框架密度和路由 token 比例体现，而不是靠每页更换完全不同的主色或装饰主题。
6. 版式系统应为后续 `scan / balanced / focus` 三类信息密度姿态预留结构空间，即使 P0 不全部暴露给用户切换，也不得把页面写死成单一密度。
7. 若首屏仍可被概括为“一个很大的壳层卡片，里面放路由入口和说明文字”，而不是“一个明确主舞台 + 一组次级入口 + 明显的后续段落”，则无论 token 和组件多精致，都不视为 `UI-V3` 达标。

## 9. 动效、响应式与可访问性

### 9.1 动效原则

1. 动效要像“仪表响应”和“场景切换”，不能像营销页炫光。
2. 页面切换、详情打开、舞台聚焦都应有明确但克制的反馈。
3. Hover、focus、selected 三种交互态必须明显区分。
4. 动效语法至少应区分 `Scene Shift`、`Instrument Response`、`Dock Open/Close`、`Reader Focus` 四类反馈，不同反馈不应共享同一套夸张位移或发光。
5. 不允许持续循环的装饰性动画成为视觉主角；任何常驻动态都必须低频、低噪音、可忽略。

### 9.2 响应式要求

1. 桌面端优先保证舞台感、分栏感和阅读 dock 能力。
2. 平板端允许弱化为单列或弱双列，但主舞台优先级不能丢。
3. 移动端允许降级为“强首屏 + 分段阅读”，不要求保留完整桌面工作台交互。
4. 小屏下详情区必须能进入明确的全屏阅读态。

### 9.3 可访问性要求

1. 不得只用颜色表达状态。
2. 背景氛围层不得影响正文对比度。
3. 键盘导航和焦点态必须保留。
4. 长 repo 名、长摘要和中英混排不能破版。

## 10. 验收标准

### 10.1 视觉换代验收

1. 用户第一眼能明确感知“这不是旧版卡片做深色皮肤，而是一套新的工作台”。
2. 页面整体更像“情报工作台”，而不是“管理后台”或“游戏官网”。
3. 五个一级视图统一但不雷同，每页都有清晰人格。
4. 全局主气质必须明确落在 `Intelligence Terminal`，其中 `Weekly` 能被识别为 `Editorial Desk`，`Knowledge Base` 能被识别为 `Research Reader`。
5. 首屏构图必须更接近现代内容型首页的组织方式：先有单一强焦点，再有次级入口，再有后续内容段落，而不是先出现多个同权入口容器。
6. 用户不需要先理解“这里有几个 panel/card”，而是能直接看出“这里的主舞台是什么、下一步看哪里”。

### 10.2 结构验收

1. `Overview` 首屏不再是等权卡片墙。
2. `Projects` 主体不再是卡片列表。
3. `Weekly` 核心趋势必须拥有主舞台，弱信号必须后置。
4. `Run Health` 必须体现诊断路径。
5. `Knowledge Base` 必须体现阅读器结构。
6. 全站必须存在稳定的 `Hero / Stage / Rail / Dock / Reader / Audit` 页面语法，而不是继续依赖一套通用 panel 平铺。
7. `Weekly` 的矩阵、轴位与证据块之间必须存在一眼可读的锚点关系。
8. 首屏不得再出现“巨型 header 卡 + 路由四宫格/四卡 + context 信息条”这种旧后台骨架；如果主舞台没有在第一屏占据绝对主导位，判定失败。

### 10.3 体验效率验收

1. 用户可在 `10-15` 秒内判断当前结果是否值得继续读。
2. 用户可在 `Projects` 中快速完成扫描并打开单个详情。
3. 用户可在 `Weekly` 中一眼抓住本周主趋势与证据矩阵的关系。
4. 用户可在 `KB` 中先读结论，再决定是否进入深层内容。
5. 用户无需先理解容器布局，就能知道“当前主舞台是什么、次级入口在哪里、继续往哪一步走”。

### 10.4 工程边界验收

1. 不破坏 `artifact-first`、时间导航、详情切换与可审计边界。
2. 不要求前端为视觉效果新增推理逻辑。
3. 新样式必须以设计 token 和稳定组件边界落地，不能继续堆叠补丁。
4. 前端输出结构必须能表达 route frame 与 surface role 语义，避免只在 CSS 里“假装有架构”。

## 11. 交付优先级

## 12. 冻结结论

`READY FOR DESIGN`

本次冻结的核心结论是：

1. `UI-V3` 必须明确摆脱卡片堆叠。
2. `pvp.qq.com` 的价值在于提供“舞台式首屏 + 强导视 + 分段前端节奏”的研究参考。
3. `Trend Radar` 必须把这种语法翻译成“可信、可读、可研究”的情报工作台，而不是做成游戏风页面。
4. 整站主气质明确以 `Intelligence Terminal` 为主，其中 `Weekly` 承接 `Editorial Desk`，`Knowledge Base` 承接 `Research Reader`。
5. 后续视觉稿、实现和回归验证都必须以这份文档为边界，不再接受“只是继续修卡片”的方向回退。
