# 需求分析：Trend Radar Visual Console

## 文档状态

- 状态：`Revised for Design`
- 需求级别：`S3 - Large Feature / System Change`
- 分析方法：依据 [需求分析 Skill](需求分析_skill.md#L1) 输出
- 本文目标：在不讨论实现方案细节的前提下，冻结 `Trend Radar Visual Console` 的产品需求、范围边界、状态语义和验收口径，作为后续设计文档与 exec-plan 的对齐基线。

## 1. Background（背景）

### 需求来源

- 当前系统已经具备 `daily report`、`run summary`、`weekly report`、`knowledge card`、`verify-daily`、`GitHub enrichment audit` 等产物。
- 这些产物在 CLI 与文件层面可用，但用户的主消费方式仍然是“翻 JSON / Markdown / 目录”。
- 产品目标已不仅是“能产出 artifact”，而是“artifact 能被稳定、连续、低心智成本地消费”。

### 业务背景

- `agent-trend-radar` 的主价值不是单纯排行，而是输出“可信、可解释、可审计”的趋势判断。
- 现有产品规格与反馈回路已经冻结了几条关键原则：
  - `rules-only` 必须可运行。
  - 高分项目必须有 evidence。
  - weekly 不能只列项目，必须给出趋势抽象。
  - 失败与降级不能被 UI 掩盖成“看起来正常”。

### 用户动机

- 用户想先判断“这次结果靠不靠谱”，再决定是否继续读内容。
- 用户想直接看到“为什么值得看”，而不是自己从多个文件拼接解释。
- 用户想从发现项目一路进入证据、历史、KB 和人工判断，不再在目录中跳转。

## 2. Goals（目标）

### 产品目标

- 把 `daily / weekly / run health / KB` 组织成统一观察与排障入口。
- 让控制台优先回答“这次结果是否可信”“今天/本周什么值得继续看”“哪里出了问题”“下一步该做什么”。
- 在不引入第二真相源的前提下，建立可持续的 artifact 消费层。

### 用户价值

- 让用户在 30 秒内完成“是否继续看本次结果”的首轮判断；若进入 UI V2.5 视觉重设计范围，则该目标收紧为 `10-20` 秒，以 [trend-radar-ui-v2-visual-redesign-requirements.md](trend-radar-ui-v2-visual-redesign-requirements.md#L1) 为准。
- 让用户在不翻底层文件的情况下完成项目级判断与周趋势判断。
- 让维护者在同一入口内完成 `warn/fail` 追因与降级识别。
- 让深度跟踪用户从趋势判断平滑进入 KB 与历史沉淀。

## 3. Users & Scenarios（用户与场景）

### 用户角色

- 技术趋势观察者：关心今天和本周有哪些 repo 值得继续看。
- AI 系统工程师：更关注 `agent runtime / memory / autonomy / MCP / infra` 等范式变化，且要求证据可追溯。
- 系统维护者：需要快速确认 pipeline 是否健康、source 是否失效、GitHub enrichment 是否降级。
- 人工 review 使用者：需要从项目或趋势进入 KB、历史 appearances 和人工判断区域。

### 使用场景

#### 场景 1：晨间查看 daily

- 触发：用户跑完 `run-daily` 后立刻查看当天结果。
- 目标：在 30 秒内判断“这次结果是否可靠”“今天最值得看的项目是什么”；若采用 UI V2.5 视觉重设计，则首轮判断目标收紧为 `10-20` 秒。

#### 场景 2：排查 `verify-daily` 告警

- 触发：`verify-daily` 返回 `warn` 或 `fail`。
- 目标：用户无需翻多个文件，就能追到 source 状态、enrichment 审计和建议动作。

#### 场景 3：查看 weekly 趋势

- 触发：用户运行 `run-weekly` 或回看最近一周结果。
- 目标：直接理解“这周强化了哪些方向、哪些只是弱信号、哪些值得下周继续跟踪”。

#### 场景 4：进入人工 review

- 触发：某个项目高分、异常增长或持续出现，用户决定进一步审查。
- 目标：从控制台直接进入项目详情、KB 卡片、历史 appearances 和人工判断区。

#### 场景 5：跨日期回看

- 触发：用户需要对比不同日期的 daily / weekly / run health 产物。
- 目标：明确当前正在消费的是哪一天或哪一个时间窗的 artifact，而不是混读“最新文件”。

### 关键路径

1. 用户进入控制台后，先确认当前日期 / 时间窗 / 运行上下文。
2. 用户先看到 run 健康、freshness 与可用性，再决定是否继续看内容。
3. 用户进入 daily 或 weekly 的重点判断层。
4. 用户从趋势或项目继续进入 evidence、risk、recommended actions、KB、历史与人工判断区。

## 4. Requirement Definition（需求定义）

### 需求 1：控制台必须先回答“本次结果靠不靠谱”

- What：控制台首页必须先展示当前 run 的总体状态、freshness、verify 结论、关键 source 状态和主要风险。
- Why：用户的第一判断不是“分数高不高”，而是“这次结果值不值得信”。
- Who：所有控制台用户。
- When：打开任意一次 daily 或 weekly 对应结果时。
- Expected outcome：用户无需先读项目列表，即可判断是否应该继续消费本次结果。

### 需求 2：控制台必须把项目判断组织成“结论优先、证据可追”

- What：项目列表与项目详情必须直接展示 `score / confidence / paradigm / evidence / risk / persistence / next actions` 等关键信息，并允许继续追到更细证据。
- Why：用户需要先理解“为什么值得看”与“为什么要谨慎”，而不是只看到一组分数。
- Who：趋势扫描用户、AI 系统工程师、人工 review 用户。
- When：浏览 daily 项目列表或单项目详情时。
- Expected outcome：用户在不翻 JSON 的情况下，也能完成项目级判断。

### 需求 3：控制台必须提供 run health 排障工作台

- What：控制台必须把 `run summary / verify-daily / source status / GitHub enrichment audit / recommended actions` 组织为统一排障视图。
- Why：当前告警、审计与建议动作分散，用户无法快速定位失败来源。
- Who：系统维护者、可用性审查用户。
- When：run 出现 `warn/fail`、source 空结果、source 失败或 enrichment 降级时。
- Expected outcome：用户能从统一入口追到问题来源、影响范围和建议动作。

### 需求 4：控制台范围必须显式覆盖 weekly 的最小消费入口

- What：控制台必须支持 weekly 的最小可消费视图，至少让用户能读取“本周总判断、核心趋势、弱信号、是否值得下周继续跟踪”。
- Why：产品目标已明确包含 weekly；如果控制台只服务 daily，会形成新的消费断层。
- Who：趋势观察者、决策用户、长期跟踪用户。
- When：查看 weekly 结果时。
- Expected outcome：用户不再需要跳回 markdown 文件才能理解 weekly 抽象结果。

### 需求 5：weekly 必须支持“无趋势周”和“弱信号周”的真实语义

- What：当最近 7 天不足以支撑稳定趋势时，控制台必须允许 weekly 呈现“尚未形成稳定趋势，仅有弱信号/待观察方向”，不得伪装成已有核心趋势。
- Why：weekly 的价值在于趋势抽象，不在于把单日热度包装成周趋势。
- Who：趋势观察者、决策用户。
- When：weekly 输入缺少重复证据、多源支撑或趋势收敛时。
- Expected outcome：用户可以区分“本周没有稳定趋势”和“系统忘了展示趋势”。

### 需求 6：控制台必须提供 KB / review 的连续入口

- What：控制台必须支持浏览 KB 列表、打开单卡片、区分机器区与人工区，并能从 daily 项目与 weekly 趋势进入对应 KB 或历史沉淀。
- Why：控制台不应只停在“发现层”，还要接住后续 review 与沉淀。
- Who：深度跟踪用户、人工 review 用户。
- When：项目值得持续关注或需要进一步审查时。
- Expected outcome：从“看到项目/趋势”到“进入沉淀和人工判断”形成连续路径。

### 需求 7：控制台必须显式绑定日期 / 时间窗 / artifact 上下文

- What：控制台必须明确当前消费的是哪一天的 daily / run health、哪一个时间窗的 weekly，以及本次视图对应的 artifact 时间戳与生成上下文。
- Why：如果没有明确上下文，用户会把不同日期或不同窗口的 artifact 混读，破坏判断可信度。
- Who：所有回看历史结果的用户。
- When：切换日期、打开历史结果、查看 weekly 时间窗或使用 `latest` 快捷入口时。
- Expected outcome：用户始终知道当前视图到底在读哪一次结果。

### 需求 8：控制台必须坚持 artifact-first，不得引入第二真相源

- What：控制台不得重算 scoring、freshness、normalize 或趋势结论；它只能消费已有 artifact，并做轻量展示聚合与可追溯关联。
- Why：现有系统的核心价值之一是规则可解释、产物可审计；新增 UI 不能破坏这一点。
- Who：所有用户，以及后续设计 / 实现人员。
- When：定义数据来源、状态口径和页面行为时。
- Expected outcome：用户在 UI 和 CLI 中看到的是同一套事实，而不是两套可能漂移的结果。

### 需求 9：控制台必须冻结失败、降级、空态的状态语义

- What：控制台必须显式区分并展示以下状态语义：
  - `无结果`：运行成功，相关 source 或视图范围内没有匹配内容。
  - `数据失败`：必须依赖的 source 或 artifact 获取失败，不能推断为“没有内容”。
  - `结果过期`：artifact 存在，但不满足当前日期 / 时间窗的新鲜度要求。
  - `降级可用`：结果可读，但部分来源、增强层或指标使用了 fallback。
  - `不可用于判断`：缺少核心证据或关键上下文，用户不应据此做趋势判断。
- Why：UI 不能把“空”“坏”“旧”“降级”混成一个视觉结果，否则会制造伪可靠感。
- Who：所有用户。
- When：任意 artifact、source、weekly 视图、项目详情或 KB 入口不完整时。
- Expected outcome：用户能区分“没有内容”和“结果不可靠”。

### 需求 10：控制台必须对 rules-only / enhancement fallback / 审计上下文可见

- What：控制台必须让用户看见本次结果是否处于 `rules-only`、增强层 fallback、部分 source 缺失、GitHub enrichment 降级等运行模式或审计上下文。
- Why：同样的项目判断，在不同运行模式下，可信度与阅读方式不同。
- Who：所有需要判断可用性的用户。
- When：daily、weekly、run health 首页以及项目详情中存在显著影响判断的模式变化时。
- Expected outcome：用户知道本次结果是“完整增强结果”还是“规则保底结果”，不会误把降级结果当完整结果消费。

### 需求 11：控制台必须保证跨视图语义一致与可钻取

- What：Overview、Projects、Run Health、Weekly、KB 中对同一 run、同一项目、同一趋势的核心结论必须一致，并支持从趋势到 supporting projects、从项目到 evidence / history / KB 的连续钻取。
- Why：如果不同视图口径不一致，控制台会比“翻文件”更难信任。
- Who：所有用户，尤其是需要在多个视图间切换的用户。
- When：从概览进入项目、从 weekly 进入趋势支撑项目、从项目进入 KB 时。
- Expected outcome：用户不会在不同视图看到相互冲突的判断或失去上下文。

## 5. Interpretation Options（多义解读）

### 诉求 A：“做一个可视化控制台”

- 可能含义 1：做一个更好看的 daily 页面。
- 可能含义 2：做一个跨 daily、weekly、run health、KB 的统一消费层。
- 本次需求按“含义 2”为准；“更好看”不是主目标，“更可判断、更可追因、更可持续消费”才是主目标。

### 诉求 B：“让用户快速判断今天值不值得看”

- 可能含义 1：优先展示高分项目。
- 可能含义 2：优先展示结果是否可信，再展示哪些项目值得继续看。
- 本次需求按“含义 2”为准，禁止把“高分”误等价为“可信”。

### 诉求 C：“支持 weekly”

- 可能含义 1：只是给一个 weekly 文件入口。
- 可能含义 2：把 weekly 当成完整消费对象，保留总判断、核心趋势、弱信号、无趋势周等完整语义。
- 本次需求按“含义 2”为准。

### 诉求 D：“支持人工 review”

- 可能含义 1：第一版直接做复杂编辑、多用户协作。
- 可能含义 2：第一版先冻结 review 入口连续性、机器区/人工区边界与历史可读性。
- 本次需求按“含义 2”为准。

### 诉求 E：“artifact-first”

- 可能含义 1：前端直接读文件即可，不必定义产品级数据口径。
- 可能含义 2：无论通过什么读取层，控制台都必须只消费既有 artifact，不得新增第二业务真相或重算核心判断。
- 本次需求按“含义 2”为准。

## 6. Constraints（约束）

### 权限规则

- 第一版以只读消费为主，不把控制台定义成新的运行调度入口。
- 第一版不要求引入多用户协作、复杂权限系统或在线写回权限。
- 如后续支持编辑，优先限定在 KB 人工区，且不得覆盖机器区。

### 数据归属

- `data/` 目录下的 `reports / scores / kb / raw / classifications` 等现有产物仍是主真相源。
- UI 层不得生成新的业务事实字段替代已有 artifact 结论。
- UI 可做轻量聚合与链接组织，但不得在用户不可见处改写原有判断口径。

### 业务规则

- 健康状态与可信度必须先于排行榜展示。
- 证据、风险、recommended actions 是一等信息，不得降格为附属说明。
- weekly 的最小消费能力不得缺席，且必须保留“核心趋势 / 弱信号 / 无趋势周”语义。
- “支持人工 review”在本轮冻结为“支持进入和查看”，不冻结复杂编辑与协作。
- 当 weekly 缺乏稳定证据时，必须明确表达“尚未形成稳定趋势”，不得伪装成核心趋势周。

### 兼容性

- 不改变现有 CLI 命令名、artifact 生成职责和 `rules-only` 主链路。
- 控制台必须兼容 `llm.enabled=false`、GitHub token 缺失、GitHub enrichment fallback 等降级形态。
- `latest` 只能作为快捷入口，不能替代明确日期 / 时间窗语义。

## 7. Acceptance Criteria（验收标准）

### 验收 1：可信度优先

- 可观测结果：用户打开控制台后，首屏即可看到 run 状态、freshness、verify 结论、关键 source 状态和主要风险。
- 成功条件：用户无需先读项目列表或底层文件，就能判断本次结果是否可靠。
- 失败条件：首页先展示榜单，而健康、freshness 或 verify 结论需要继续翻页后才能看到。

### 验收 2：项目判断可直接消费

- 可观测结果：项目列表和详情中直接可见 `score + confidence + paradigm + top evidence + risk + persistence + next actions`。
- 成功条件：用户不看 JSON 也能判断项目为什么值得看、为什么需要谨慎看。
- 失败条件：UI 退化成排行榜或资讯流，关键证据和风险需要反复跳文件才能看到。

### 验收 3：Run Health 排障成立

- 可观测结果：用户能在同一视图内看到 verify 结果、source 状态、GitHub enrichment 审计和 recommended actions。
- 成功条件：遇到 `warn/fail`、`source empty`、`source failed`、`fallback` 时，用户可以沿视图直接追到来源与动作建议。
- 失败条件：告警与审计仍分散在多个文件，控制台没有形成闭环排障路径。

### 验收 4：weekly 最小消费入口成立

- 可观测结果：控制台中存在 weekly 的独立入口或等价视图，能消费周报总判断、核心趋势、弱信号以及下周跟踪建议。
- 成功条件：用户不必离开控制台也能理解“本周趋势是什么”。
- 失败条件：控制台只覆盖 daily，weekly 仍需回到 markdown 或目录手动查找。

### 验收 5：无趋势周与弱信号周语义成立

- 可观测结果：当 weekly 缺少稳定趋势证据时，页面会明确显示“尚未形成稳定趋势”或“当前以弱信号为主”。
- 成功条件：用户能区分“本周确实没有稳定趋势”和“控制台丢了 weekly 内容”。
- 失败条件：系统在证据不足时仍强行产出看似确定的核心趋势卡片。

### 验收 6：KB / review 路径成立

- 可观测结果：用户能从项目或趋势进入 KB，看到机器区与人工区，并浏览历史沉淀。
- 成功条件：`发现项目 / 趋势 -> 进入沉淀 -> 继续人工判断` 形成连续路径。
- 失败条件：控制台只能展示发现层，无法进入 KB，或无法区分机器区与人工区。

### 验收 7：日期 / 时间窗上下文明确

- 可观测结果：当前视图明确标识所消费的 daily 日期、weekly 时间窗、artifact 时间戳与运行上下文。
- 成功条件：用户不会把不同日期或不同窗口的 artifact 混读成同一次结果。
- 失败条件：UI 默认只展示“最新”，但用户无法确认它具体对应哪一天、哪一周或哪一次运行。

### 验收 8：不引入第二真相源

- 可观测结果：UI 中所有核心状态和结论都可追溯到已有 artifact。
- 成功条件：同一日期 / 时间窗下，CLI 与 UI 对核心状态的解释保持一致。
- 失败条件：UI 私自重算或改写核心判定，导致与 artifact 语义漂移。

### 验收 9：失败、降级、空态可区分

- 可观测结果：用户能区分 `无结果 / 数据失败 / 结果过期 / 降级可用 / 不可用于判断`。
- 成功条件：降级与空态是显式语义，而不是隐藏在空白页、默认文案或看似正常的卡片里。
- 失败条件：页面只显示“没内容”或“看起来正常”，用户无法判断是没有结果还是结果失效。

### 验收 10：审计上下文可见

- 可观测结果：用户可以看见本次结果是否是 `rules-only`、增强 fallback、source 缺失或 enrichment 降级运行。
- 成功条件：用户知道当前结果属于完整结果、保底结果还是降级结果。
- 失败条件：运行模式变化影响判断，但 UI 没有显式告知用户。

### 验收 11：跨视图一致且可钻取

- 可观测结果：Overview、Projects、Weekly、Run Health、KB 对同一 run / 项目 / 趋势的核心判断一致，且支持连续钻取。
- 成功条件：用户从 weekly 趋势进入 supporting projects、从项目进入 evidence / KB / history 时不会丢失上下文或看到冲突结论。
- 失败条件：不同视图对同一对象口径不一致，或钻取后只能看到脱节信息。
