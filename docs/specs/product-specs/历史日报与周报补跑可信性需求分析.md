# 历史日报与周报补跑可信性需求分析

## 文档状态

- 状态：`Frozen for Design`
- 需求级别：`S2 - Medium Feature`
- 本文目标：冻结“历史日期的 daily / weekly backfill 是否真正基于目标日期证据生成，而不是把当前数据伪装成历史结果”的产品需求。

## 1. 输入研读结论

### 目标

- 让用户在执行 `run-daily --date YYYY-MM-DD` 时，得到一份对该目标日期语义真实的日报。
- 让 `run-weekly --backfill-missing-days` 依赖的缺失日补跑结果具备可审计的历史真实性，而不是只满足“文件补齐了”。
- 让系统区分“真历史数据缺失”与“可以继续生成看起来正常的产物”这两件事。
- 让用户在历史回放场景下明确知道：哪些 source 是精确历史、哪些是有界历史、哪些只能作为推断或上下文。

### 用户

- 回补遗漏日报的维护者。
- 追溯某一周趋势判断是否成立的分析者。
- 需要审计“这份历史日报到底能不能信”的技术决策者。

### 当前痛点

- `--date` 已经传到了 pipeline，但部分 source 仍可能读取当前实时结果，再把时间戳或 `effective_date` 标成目标日。
- 历史 daily 的 `run-summary` 和 `verify-daily` 目前更多是在自证字段一致，而不是独立证明 source 证据确实来自目标日期。
- weekly 缺失日补跑目前只验证 daily artifact 是否存在且结构兼容，无法判断该 daily 是否属于“历史语义真实”。
- 对“没有历史数据”与“历史数据不存在但可以用今天的数据凑一个结果”之间的边界缺少产品级约束。

### 触发条件

- 用户手动执行历史日期 daily：`run-daily -- --date <past-date>`
- 用户执行历史周窗口：`run-weekly -- --date <past-date>`
- 用户执行 `run-weekly -- --backfill-missing-days`
- 用户通过 visual console 或 report 回看历史日报/周报

### 已知约束

- `agent-trend-radar-product-spec.md` 已冻结：daily 主价值是“当天值得关注的新鲜项目”，历史内容不能伪装成当天主榜单。
- `daily-report-freshness-readability-design.md` 已冻结：历史 fallback 可以存在，但必须显式降级，不能继续装作当天新鲜发现。
- 当前 source 历史能力并不对称：`agents-radar` 有按日期 digest；Trendshift、GitHub Trending、watchlist、live star delta 的历史能力需要额外设计。
- 不允许通过 LLM 或 report 文案掩盖 source 本身的历史证据缺失。

## 2. 需求规模判断

本需求判定为 `S2 - Medium Feature`。

原因：

- 涉及 CLI、Signal、storage、report、verification、weekly window 五层契约。
- 会改变历史 daily / weekly 的产品语义与验收方式。
- 需要新增长期归档能力与 source 能力分级，但不要求本轮就把所有过去日期补齐。

## 3. 多义解释

### 诉求 A：历史日报要“是真的”

可能含义 1：
用户希望“报告标题和文件名是历史日期”。

可能含义 2：
用户希望“报告中的主要证据确实来自目标日期，至少不能偷偷混入今天的实时源”。

本需求按“含义 2”为主处理；含义 1 只是表面形式，不是成功标准。

### 诉求 B：周报缺失日补跑要可依赖

可能含义 1：
只要补齐 daily 文件，weekly 就能继续跑。

可能含义 2：
weekly 依赖的每个 daily 都要达到最小可信门槛，否则 weekly 结论不应继续输出或必须显式降级。

本需求按“含义 2”为主处理。

### 诉求 C：真正的历史数据拿不到时怎么办

可能含义 1：
系统应该继续生成“看起来像历史日报”的内容，尽量不要空。

可能含义 2：
系统应该明确告诉用户哪些 source 没法为该日期提供真实历史证据，并拒绝把它们包装成精确历史。

本需求按“含义 2”为主处理。

## 4. 用户与场景建模

### 用户角色

- 历史补跑维护者：关注补跑是否真实、可重复、可自动化。
- 趋势审计用户：关注某一天或某一周的趋势结论是否可以复核。
- 回看用户：关注 visual console / daily / weekly 在历史日期上的语义是否稳定。

### 使用场景

#### 场景 1：补跑一条漏掉的历史日报

- 触发：用户执行 `run-daily -- --date 2026-04-27`
- 目标：获得一份可解释“这份日报对 2026-04-27 到底有多真实”的结果

#### 场景 2：周报自动补缺

- 触发：用户执行 `run-weekly -- --date 2026-05-03 --backfill-missing-days`
- 目标：系统先补缺日，再只在 daily 可信度满足要求时生成 weekly

#### 场景 3：历史回看

- 触发：用户在 visual console 或报告目录中回看历史日期
- 目标：看到的不是“后来补写但被伪装成当天 fresh 的结果”

## 5. 产品定义

### 需求 1：历史真实性优先于补齐率

- What：历史 daily / weekly 的首要目标是“对目标日期语义真实”，不是“尽量生成非空结果”。
- Why：如果系统为了补齐率继续使用当天数据伪装成历史结果，会直接破坏回放和审计价值。
- Who：所有历史补跑与历史回看用户。
- When：任何 `date < current_local_date` 的 daily / weekly 运行。
- Expected outcome：系统宁可显式标记“该 source 无法提供真实历史”，也不允许假装精确命中目标日期。

### 需求 2：source 历史能力必须分级可见

- What：每个 source 都必须暴露其历史能力等级，而不是统一用一个 `effective_date` 代表“历史可信”。
- Why：不同 source 的真实历史能力不同，产品必须让用户和下游 weekly 知道差异。
- Who：历史补跑用户、weekly 消费方、验证层。
- When：生成 daily / weekly artifact 与 verify 结果时。
- Expected outcome：用户能区分 `精确历史`、`有界历史`、`推断历史`、`不可用历史`。

### 需求 3：历史运行与当天实时运行必须语义分离

- What：历史 replay 不能继续复用“fresh_today”这类只适用于当天实时扫描的表达。
- Why：目标日与执行日不同，若仍沿用当天语义，用户会把“对目标日期精确”误读成“对执行当天新鲜”。
- Who：所有 daily / weekly 读者。
- When：历史 daily、历史 weekly、run summary、verify-daily、visual console。
- Expected outcome：历史 artifact 显式展示它是 `historical replay`，并使用针对 replay 的状态语义。

### 需求 4：从现在开始构建可回放归档

- What：对没有官方历史读取能力的 source，系统必须从当前开始保存可回放归档与空结果观察回执。
- Why：很多 source 的过去不能凭空补齐，但从现在开始可以逐日变成真历史。
- Who：未来所有历史补跑用户。
- When：每天 live run 成功或失败时。
- Expected outcome：未来的历史补跑越来越依赖归档，而不是依赖“重新抓当天网页”。

### 需求 5：缺历史证据时必须给出明确降级而非偷偷补位

- What：如果某 source 对目标日缺少真实历史证据，系统只能：
  - 标为不可用
  - 标为推断/代理
  - 或降级为上下文
- Why：用户需要知道“缺证据”本身就是结果。
- Who：历史补跑与 weekly 生成用户。
- When：daily / weekly 计算主榜单与趋势卡片时。
- Expected outcome：不得用今天抓到的 live 结果伪装成历史 `fresh`。

### 需求 6：weekly 只能消费历史语义可信的 daily

- What：weekly 对缺失日的补跑，不只要求文件存在，还要求 daily 达到最小 replay 可信门槛。
- Why：weekly 趋势比 single-day 报告更依赖跨天语义一致性。
- Who：weekly 消费方。
- When：weekly window 检查和 backfill 后复检。
- Expected outcome：若某日 daily 仍包含不可接受的伪历史主榜单，weekly 必须失败或显式降级。

### 需求 7：验证层必须独立证明历史证据，不得只信 report 自述

- What：`verify-daily` / 后续 weekly audit 必须读取 source 侧归档或元数据，独立验证历史来源。
- Why：不能让 report 自己写什么，verify 就信什么。
- Who：审计用户、自动验证脚本。
- When：历史 daily / weekly 验证。
- Expected outcome：验证失败时能指出具体是哪个 source、哪类证据缺失。

## 6. 业务规则与约束

### 历史能力分级规则

用户可见层必须支持以下最小语义：

- `精确历史`
  定义：source 证据直接来自目标日期的官方历史接口、事件日志、已归档快照或同日 digest。
- `有界历史`
  定义：source 可以在有限时间窗内提供真实历史，例如仅最近 30 天事件、或依赖连续日快照。
- `推断历史`
  定义：source 只能根据其它证据重建代理信号，不能当作原始历史事实。
- `不可用历史`
  定义：source 对目标日没有足够证据，不能参与历史主榜单判断。

### 历史日报主榜单规则

- 历史 daily 的主榜单只能使用 `精确历史` 或设计明确允许的 `有界历史` 信号。
- `推断历史` 只能进入补充观察或单独标记的代理区块。
- `不可用历史` source 不得进入主榜单。
- 任何 source 只因重新执行 live fetch 得到结果，不得被标记为历史精确命中。

### 空结果证明规则

- 对“当天没有命中”的结论，系统必须保留该 source 的观察回执。
- 缺少观察回执时，系统只能说“未持有该日证据”，不能说“该日没有事件/没有候选”。

### 来源边界规则

- `agents-radar`：若存在目标日 digest，则可作为精确历史上下文与候选来源。
- `Trendshift`：只有目标日归档快照或官方历史接口结果，才能算精确历史；重新抓当前页面不算。
- `GitHub Trending`：只有目标日归档榜单或等价官方历史记录，才能算精确历史；Search API 重跑结果不算。
- `watchlist live activity`：只有目标日事件日志或当日观察回执，才能对“有/无活动”作出历史结论。
- `github_live_star_delta`：只有目标日可审计 star events 或相邻快照链条，才能作为历史日增事实。

### 当天实时模式保护

- 本需求不改变 `date == current_local_date` 的 live 运行目标：当天 daily 仍应优先消费当次实时发现层。
- 但 live 运行也必须同步写入未来可回放的归档与观察回执。

## 7. 验收口径

### 需求 1：历史真实性优先于补齐率

- 可观察结果：
  历史 daily 在 source 无真实历史证据时显式降级或失败，不再继续产出伪精确主榜单。
- 成功条件：
  系统不会再把当前 live 结果包装成目标日期 exact/fresh 结果。
- 失败条件：
  历史 report 仍主要靠执行当天抓取的数据支撑，并被展示成目标日主榜单。

### 需求 2：source 历史能力必须分级可见

- 可观察结果：
  每个 source 都能被标记为 `精确历史 / 有界历史 / 推断历史 / 不可用历史` 之一。
- 成功条件：
  用户与 weekly 消费方可以根据分级判断可信度。
- 失败条件：
  仍只有一个模糊的 `effective_date` 或单一 freshness 标签。

### 需求 3：历史运行与当天实时运行必须语义分离

- 可观察结果：
  历史 artifact 显式表明自己是 `historical replay`，并使用专门的历史状态表达。
- 成功条件：
  用户不会把历史 replay 的“精确命中目标日期”误读成“对执行当天新鲜”。
- 失败条件：
  历史 replay 仍继续输出 `fresh_today` 一类 live 语义。

### 需求 4：从现在开始构建可回放归档

- 可观察结果：
  live run 会为支持归档的 source 写入日期化快照、元数据和空结果观察回执。
- 成功条件：
  新日期在未来执行历史 replay 时可直接依赖归档。
- 失败条件：
  重新抓 live 页面仍是唯一历史回放手段。

### 需求 5：缺历史证据时必须给出明确降级而非偷偷补位

- 可观察结果：
  缺证据 source 会在 report / verify 中被单独指出。
- 成功条件：
  推断或缺失 source 不会继续占据历史主榜单同等优先级。
- 失败条件：
  报告非空但用户无法知道哪些部分其实没有历史证据。

### 需求 6：weekly 只能消费历史语义可信的 daily

- 可观察结果：
  weekly window 检查不仅看文件存在，也看 replay 可信门槛。
- 成功条件：
  weekly 不会吞下伪历史 daily 继续产出看似稳定的趋势结论。
- 失败条件：
  只要补齐 daily 文件，weekly 就静默继续。

### 需求 7：验证层必须独立证明历史证据

- 可观察结果：
  `verify-daily` 能指出 source 对应的归档/事件/观察回执是否存在，以及是否与 report 自述一致。
- 成功条件：
  验证不再仅依赖 report 内部字段自证。
- 失败条件：
  report 写什么 verify 就信什么。

## 8. 需求冻结检查

当前已经冻结的内容：

- 历史 daily / weekly 的第一原则是语义真实，不是补齐率。
- source 历史能力必须对用户与 weekly 消费层可见。
- 历史 replay 与当天 live 必须使用不同的状态表达。
- 没有历史证据时必须显式降级、标记或失败。
- weekly backfill 必须建立在可信 daily 之上。
- verify 层必须读取独立历史证据。
- 从当前起，系统必须开始积累未来可回放的归档与观察回执。

仍未冻结、需要设计阶段做出明确决策的问题：

1. 对 `有界历史`，哪些 source 可以进入历史主榜单，哪些只能进入补充观察？
2. weekly 对历史 daily 的最小可信门槛，采用“失败即阻断”还是“生成但显式降级”的具体策略。
