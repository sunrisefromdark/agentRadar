# 外部发现与补证信号层需求分析

## 文档状态
- 状态：`Frozen for Design`
- 需求级别：`S3 - Large Feature / System Change`
- 分析方法：依据 [需求分析_skill.md](需求分析_skill.md)
- 本文目标：在不替代现有 GitHub / Trendshift 主判断链路的前提下，冻结“外部发现与补证信号层”的产品需求边界，用于支持更早发现 agent 相关候选，并为已有候选补充外部讨论证据。

## Requirement Size
`S3 - Large Feature / System Change`

判定原因：
- 这不是局部字段扩展，而是新增一层新的产品能力边界：系统从“主源发现与判断”扩展为“主源判断 + 外部次级判断信号”。
- 该需求同时影响 daily、weekly、候选发现、项目解释、趋势判断和用户对结果可信度的理解。
- 该需求跨越 `Signal -> Normalize -> Filter -> Action` 的多层语义，但必须保持现有主链路的稳定性、可降级性和可审计性。

## 1. Background

### 需求来源
- 用户明确希望在现有 GitHub / Trendshift 之外，引入 Twitter、Reddit 以及网页搜索等外部渠道，用于更早发现 agent 相关项目、论文、工具和讨论。
- 用户同时确认这些新增信号可以参与 daily / weekly 判断，但权重必须低于 GitHub / Trendshift。
- 用户已经接受“分层型”方向：外部信号进入正式判断链路，但只作为次级判断信号，而不是新的主裁决链路。

### 业务背景
- 当前系统已经具备一套以 GitHub、Trendshift 和已提交 artifacts 为核心的 explainable 主链路，优势在于可回放、可解释、可降级。
- 当前系统的不足不在于“完全没有发现能力”，而在于 GitHub 之外的生态变化进入得不够早，且“谁在讨论、讨论是否持续、是否为头部人群在讨论”这类判断依据仍然偏弱。
- 社媒和外部搜索类渠道的价值主要体现在前瞻性和讨论语境，但天然也更高噪声、更高漂移、更难复现。

### 用户动机
- 用户希望更早看到尚未在 GitHub 指标上充分显现、但已经在核心开发者和研究机构圈层传播的新项目和新方向。
- 用户希望系统不仅能说“今天什么热”，还能说“为什么值得关注”“是不是头部人群在讨论”“这波讨论是偶发还是持续”。
- 用户希望 weekly 输出更接近趋势判断，而不只是对 GitHub 项目的结构化汇总。

## 2. Goals

### 产品目标
- 新增一层 `external discovery & evidence` 产品能力，用于承接外部发现信号和外部补证信号。
- 保持 GitHub / Trendshift 作为主判断链路不变，外部层只作为次级判断信号参与 daily / weekly。
- 让系统更早发现新候选，同时提升已有候选的讨论质量判断、权威性判断和趋势确认能力。
- 在外部信号不可用、限流或质量不足时，系统仍然能够稳定退回现有主链路产出。

### 用户价值
- 用户可以更早发现 agent 相关的新项目、新论文、新工具和新讨论。
- 用户可以看到“是谁在讨论、讨论强度如何、是否跨平台、是否持续”，而不再只依赖 GitHub 侧热度判断。
- 用户可以在 weekly 中更容易理解一个方向是否真的在形成趋势，而不是只看到一组项目列表。

## 3. Users & Scenarios

### 用户角色
- 趋势扫描型用户：希望尽早知道 agent 生态里刚冒头的新候选和新方向。
- 深度判断型用户：希望在已有候选上获得更强的“值得不值得跟”的证据。
- 生态研究型用户：更关心核心机构、知名开发者、科研团队和极客圈层正在推动什么。
- watchlist 用户：希望知道自己长期关注的对象是否开始获得外部讨论放大。

### 使用场景

#### 场景 1：更早发现新候选
- 触发：外部核心实体首次提到一个新 repo、论文、demo 或工具。
- 目标：系统在 GitHub 热度完全起量前，先把该对象识别为值得观察的候选。

#### 场景 2：为已有候选补证
- 触发：某个已进入候选池的项目，在 Twitter / Reddit / Web 搜索结果中被多个独立实体提到。
- 目标：系统补充“谁在讨论、讨论是否持续、是否跨平台”的次级判断依据。

#### 场景 3：判断讨论是否只是短暂噪声
- 触发：项目在单平台出现高热，但缺乏跨平台确认或持续性。
- 目标：系统避免把单次爆发误判为已形成趋势。

#### 场景 4：weekly 抽象趋势时需要外部确认
- 触发：某一方向在 GitHub 信号之外，也开始被机构、研究者或开发者持续讨论。
- 目标：weekly 能更有把握地判断“这个方向正在强化”，而不是只看到项目个体。

### 触发条件
- 新的一日 daily 信号收集开始。
- weekly 汇总时需要对趋势方向补充讨论证据。
- 已有候选项目进入重点观察或接近趋势边界。
- 核心实体或观察列表实体在外部渠道产生新的 agent 相关发布或讨论。

### 关键用户路径
1. 系统先完成现有主链路的采集、标准化、评分和基础判断。
2. 外部层补充发现与讨论证据，并尝试关联到已有候选或生成新的观察候选。
3. daily 在基础结果之上增加“新发现了什么”和“哪些已有候选获得了外部补证”。
4. weekly 在基础项目汇总之上增加“外部讨论是否支持某个趋势方向正在强化”的判断依据。

## 4. Requirement Definition

### 需求 1：系统必须支持“外部发现信号”与“外部补证信号”两类不同语义
- What：系统必须把外部层明确分为“发现型目标”与“判断型目标”两类信号。
- Why：用户同时需要更早发现新候选，也需要为已有候选补充更强的讨论依据；两者不是同一种产品语义。
- Who：daily / weekly 消费者、趋势研究用户、watchlist 用户。
- When：任何外部渠道信号进入正式消费链路时。
- Expected outcome：用户能够区分“这是一个新的观察候选”与“这是对已有候选的额外证据”。

### 需求 2：外部层必须作为次级判断信号参与 daily / weekly，而不能替代主源
- What：外部层必须进入正式 daily / weekly 判断链路，但其权重和裁决权限必须低于 GitHub / Trendshift。
- Why：用户已经确认要让外部信号参与正式判断，但系统现有可信度依赖主源稳定性；外部层不能破坏这一点。
- Who：所有 daily / weekly 用户。
- When：daily / weekly 正式输出时。
- Expected outcome：用户能看到外部信号确实影响候选发现和趋势判断，但不会出现“纯靠社媒热度就主导结论”的结果。

### 需求 3：系统必须支持“谁在讨论”的权威性判断
- What：外部层必须能表达讨论者身份层级，例如机构、研究团队、知名开发者、技术精英、普通账号；且 V1 必须同时覆盖机构 / 团队 / 个人三类主体，而不是只覆盖机构账号。
- Why：用户明确重视“是不是头部人群在讨论”，这直接影响讨论价值判断。
- Who：深度判断型用户、weekly 用户。
- When：已有候选获得外部讨论补证时。
- Expected outcome：用户不只看到“有讨论”，还能看到“讨论来自什么层级的人群”，以及这些讨论者是否属于已维护的头部实体名单。

### 需求 4：系统必须支持“讨论强度与持续性”的判断
- What：外部层必须支持表达讨论数量、独立讨论者数量、持续天数、跨平台重复出现等信号；V1 的最小统计口径必须冻结为“底层按事件采集、分析按日聚合、weekly 按最近 7 日窗口表达”。
- Why：单次爆发与持续发酵的意义不同；用户希望系统能识别真正有延续性的变化。
- Who：趋势研究用户、weekly 用户。
- When：daily 识别观察候选、weekly 判断趋势强化时。
- Expected outcome：用户能够判断某个项目或方向是短促噪声，还是在持续积累关注。

### 需求 5：系统必须支持“跨平台确认”语义
- What：当同一项目、论文或方向在多个外部平台被独立提及，系统必须能够把它表达为更高质量的次级判断信号。
- Why：跨平台确认比单平台热度更接近真实外部共识。
- Who：daily / weekly 用户。
- When：外部讨论被聚合并进入正式判断时。
- Expected outcome：用户能看到哪些讨论已经越过单一平台噪声，形成更稳定的外部关注。

### 需求 6：系统必须允许生成“观察候选”，但不能让外部层直接制造高置信主结论
- What：外部层可以引入新的观察候选、待跟踪方向或待补证对象，但不能单独把对象抬升为主结论、主趋势或高置信推荐。
- Why：外部层的核心价值是前瞻发现，不是替代主源完成成熟判断。
- Who：趋势扫描型用户、weekly 用户。
- When：外部层发现 GitHub 尚未充分显现的新项目或新方向时。
- Expected outcome：用户更早看到值得观察的线索，但不会把早期噪声误读为已经成立的趋势。

### 需求 7：系统必须能把外部讨论关联到“项目级”和“方向级”两种对象
- What：外部层必须同时支持把信号关联到具体 repo / paper / product，也支持关联到尚未完全收敛成单一项目的方向性主题；当方向讨论暂时无法关联到明确 repo / paper 时，V1 必须允许其以“方向级观察信号”进入 weekly，而不是强行落到项目级。
- Why：用户既关心具体项目，也关心“memory”“runtime”“skills-tools-mcp”这类方向何时在外部讨论层面开始强化。
- Who：daily 用户、weekly 用户、生态研究用户。
- When：外部信号进入聚合与判断阶段时。
- Expected outcome：daily 更利于项目级消费，weekly 更利于方向级抽象，而两者都能共享外部证据；无法绑定明确对象的方向讨论也有受控的表达位置。

### 需求 8：外部层失效时系统必须可降级
- What：当外部源限流、认证失效、工具不可用、采集结果缺失或质量不足时，系统必须退回现有主链路，不得导致 daily / weekly 主产物失败或语义失真。
- Why：外部层是增强层，不是生产系统的单点依赖。
- Who：所有用户。
- When：任何外部采集异常时。
- Expected outcome：用户仍能获得稳定结果，且能感知本次外部信号是否参与了判断。

## 5. Interpretation Options

### 关键请求 A：“把 Twitter / Reddit 等作为新的信号源”
- 解释 A1：把外部平台视为新的主信号源，与 GitHub / Trendshift 同级。
- 解释 A2：把外部平台视为次级判断信号层，参与发现和补证，但不与主源同级。
- 推荐冻结：采用 `A2`
- 原因：用户已明确确认“可参与 daily / weekly，但权重低于 GitHub / Trendshift”；`A1` 会破坏当前可复现与可审计边界。

### 关键请求 B：“更早发现新项目”
- 解释 B1：让外部层直接生成正式高优先级项目推荐。
- 解释 B2：让外部层先生成观察候选、待补证对象或早期方向信号。
- 推荐冻结：采用 `B2`
- 原因：早发现的价值在于提早纳入视野，而不是跳过主源成熟过程。

### 关键请求 C：“给已有候选补充谁在讨论、讨论强度如何”
- 解释 C1：把外部讨论只作为说明文案展示，不影响判断。
- 解释 C2：把外部讨论作为正式次级判断信号，影响 watch、观察中趋势、讨论强度等判断。
- 推荐冻结：采用 `C2`
- 原因：用户希望这些信息真正参与 daily / weekly，而不是只做阅读附件。

### 关键请求 D：“大厂、技术精英、极客、科研机构的新发布和讨论”
- 解释 D1：只看机构官方账号或组织层级。
- 解释 D2：同时覆盖机构、研究团队、知名开发者和高质量技术圈层，但按层级区分权重。
- 推荐冻结：采用 `D2`
- 原因：用户明确关心的不只是机构公告，也包括技术精英和开发者圈层的前沿扩散。

### 关键请求 E：“weekly 趋势判断要用外部讨论”
- 解释 E1：外部层只支持项目级判断，weekly 不做方向抽象。
- 解释 E2：外部层既支持项目级补证，也支持方向级趋势强化判断。
- 推荐冻结：采用 `E2`
- 原因：用户目标明确包含“方向是否在形成趋势”，这要求 weekly 吃到方向级外部信号。

## 6. Constraints

### 权限与裁决边界
- 外部层不能替代 GitHub / Trendshift 成为主裁决链路。
- 外部层不能单独决定最终高置信主结论。
- 外部层不能覆盖或否认主源已经成立的 freshness 语义。
- 外部层可以增强、下调或延缓消费口径，但必须是次级判断，不是一级裁决。

### 数据归属与事实边界
- 外部层表达的是“外部讨论信号”，不是“项目客观成熟度事实”。
- 外部层中“谁在讨论”与“讨论强度”属于次级判断依据，不等价于项目质量本身。
- 没有被主源确认的新对象，只能以观察候选、待补证对象或弱趋势信号表达。
- 外部层无法稳定关联到对象时，不得伪造确定归属。

### 业务规则约束
- GitHub / Trendshift 继续回答“今天客观上什么值得看”。
- 外部层回答的是“哪些对象更早冒头了”“哪些已有候选得到了更强的外部讨论支持”。
- daily 中外部层优先服务发现与补证。
- weekly 中外部层优先服务趋势强化、跨平台确认和权威性判断。
- 外部层所有正向作用都必须有上限，防止单平台高噪声主导结果。
- V1 正式外部层平台范围必须收敛，不得在设计阶段默认继续扩展到视频、播客或长内容平台。

### 兼容性约束
- 当前无外部层时已有的 daily / weekly 产物必须继续成立。
- 外部层不可用时，系统必须退化为当前行为，而不是变成空结果或半结果。
- 外部层接入后，用户仍然要能区分主源判断与外部增强判断，不得混写成单一无来源结论。

## 7. Acceptance Criteria

### 验收 1：外部层正式参与判断，但不与主源同级
- 可观察结果：daily / weekly 中出现外部发现与外部补证语义，且用户能感知其为次级判断信号。
- 成功条件：外部层能改变观察候选、讨论强度或趋势强化表达，但不能单独形成高置信主结论。
- 失败条件：系统出现“纯靠外部平台热度就主导主结论”的结果。

### 验收 2：系统能更早发现新候选
- 可观察结果：daily 能出现“新发现候选”或“待补证候选”类型结果。
- 成功条件：用户能在 GitHub 热度完全起量前看到新的 agent 相关观察对象。
- 失败条件：外部层只能给已有对象补充说明，无法带来新的前瞻发现。

### 验收 3：系统能表达“谁在讨论”
- 可观察结果：项目或方向级结果中可见讨论者层级信息。
- 成功条件：用户能区分“头部机构在讨论”“核心团队在讨论”“知名开发者在讨论”“普通社区在讨论”，且头部判断来自已维护的 tier 名单而非临时主观描述。
- 失败条件：系统只有总提及数，没有讨论者层级语义。

### 验收 4：系统能表达“讨论强度是否持续”
- 可观察结果：结果中能体现独立讨论者数量、持续天数、跨平台出现等语义。
- 成功条件：用户能判断某对象是偶发噪声还是持续发酵，且 daily 与 weekly 在同一套“按日聚合 + 7 日窗口”口径上成立。
- 失败条件：单次高热与连续多日强化在系统中没有明显区别。

### 验收 5：weekly 能使用外部层强化趋势判断
- 可观察结果：weekly 对某些趋势方向的判断包含外部讨论强化证据。
- 成功条件：用户能理解某一方向为什么不只是 GitHub 项目聚集，而是已经得到外部讨论确认；对暂时无法关联 repo / paper 的方向，也能以“方向级观察信号”而非项目级结论表达。
- 失败条件：weekly 中外部层无法提升方向级判断，只停留在零散项目注释。

### 验收 6：外部层失效时可降级
- 可观察结果：外部采集异常时，daily / weekly 仍能产出，且不会把缺失的外部层结果伪装成正常存在。
- 成功条件：主产物继续稳定，外部层缺失被正确表达为未参与或参与不足。
- 失败条件：外部工具异常直接导致主产物失败，或结果语义混乱。

## 8. Risks & Gaps

### 风险
- 如果“发现型信号”与“判断型信号”不分层，系统会把早期弱线索误写成正式判断。
- 如果外部层与主源层边界不清，用户会误以为社媒热度与项目质量等价。
- 如果“谁在讨论”的层级不冻结，后续设计会在机构、研究团队、知名开发者和普通社区之间反复摇摆。
- 如果 weekly 只吸收项目级补证而不支持方向级聚合，外部层很难真正提升趋势判断价值。
- 如果外部层没有明确降级语义，生产系统会在外部工具漂移时出现结果理解风险。

### 已冻结边界口径
- V1 正式外部层平台冻结为 `X / Twitter`、`Reddit`、`Hacker News` 与 `官方网页 / 博客页`；其中前 3 类承担讨论与发现信号，官方网页 / 博客页承担补证与发布确认信号。
- `YouTube / Bilibili / 播客 / 微信公众号 / 长视频与长内容平台` 不属于 V1 正式外部层，只属于未来扩展或人工补证范围；设计阶段不得默认将其并入正式链路。
- GitHub 不属于本需求里的“外部层平台”，因为 GitHub 已经位于主信号体系；后续设计不得把 GitHub 再次并入外部层口径。
- 讨论强度的 V1 最小统计粒度冻结为：`底层按事件采集`、`分析按日聚合`、`weekly 按最近 7 日窗口表达`；不得只按周统计，也不得直接跳过按日层。
- “头部人群”的 V1 最小定义冻结为同时覆盖 `机构类`、`团队类`、`个人类` 三类主体；但只有进入维护名单并完成 tier 标注的实体，才可参与“头部讨论”判断。
- “头部讨论”判断必须是分层的，而不是二元的；V1 应至少支持类似 `core / proven / watch` 的层级语义，避免把所有非普通账号混成一个桶。
- 无法关联到明确 repo / paper 的外部方向讨论，V1 允许进入 weekly，但只能以“方向级观察信号”或“方向级观察卡”表达，不能直接抬升为项目级结论、高置信推荐或正式主趋势结论。
- 方向级观察信号必须显式标注“尚未绑定明确 repo / paper”，且其权重低于项目级补证信号。
- 方向级观察信号进入 weekly 前，设计上至少应要求满足若干收敛条件中的一部分，例如跨平台重复出现、多主体独立讨论、连续多日出现、或头部实体参与。

### 隐含假设
- 用户接受“外部层的价值在于前瞻与补证，而不是替代主源”这一基本心智。
- 用户愿意把“外部讨论很强”与“项目本身很成熟”视为不同维度。
- 系统未来可以维护一套稳定的实体层级，而不是完全依赖无差别关键词搜索。

## 9. Design Handoff Constraints

以下约束视为本轮需求冻结结果，后续撰写设计文档的 agent 默认必须按此展开，而不是重新打开产品边界讨论：
- 外部层是次级判断信号，不是主裁决链路。
- 外部层同时服务“发现”和“补证”两类目标。
- daily / weekly 都要消费外部层，但消费语义不同。
- 外部层必须可降级，且其权威性、持续性和跨平台性是核心判断维度。
- V1 正式平台范围默认固定为 `X / Twitter`、`Reddit`、`Hacker News`、`官方网页 / 博客页`，设计文档不得擅自扩到视频或长内容平台。
- V1 讨论强度默认采用“事件采集 -> 按日聚合 -> weekly 7 日窗口”的单一统计口径，设计文档不得自行改成纯按周口径。
- V1 “头部人群”默认覆盖机构 / 团队 / 个人三类主体，并基于维护名单和 tier 分层判断；设计文档不得收缩为“仅机构”。
- V1 必须为“无法绑定 repo / paper 的方向讨论”保留受控表达位，但该表达位只能是方向级观察信号，不得提升为项目级强结论。
- 如果设计阶段确实发现上述冻结口径无法实现，必须显式回到需求阶段重新讨论，而不是在设计文档中隐式改写。

## 10. Requirement Freeze Status

`READY`

满足原因：
- 需求边界清晰：外部层正式进入判断链路，但严格低于 GitHub / Trendshift。
- 用户价值清晰：更早发现候选、补充讨论语境、提升趋势判断质量。
- 语义分层清晰：发现信号与补证信号已区分，项目级与方向级消费边界已冻结。
- V1 边界清晰：平台范围、讨论强度口径、头部人群定义与方向级观察信号表达均已冻结。
- 约束清晰：主源优先、外部层次级、结果可降级、用户必须可感知来源层级。
- 验收清晰：可观察输出、成功条件、失败条件均已定义。

可直接进入下一阶段设计：
- 设计外部层与现有 daily / weekly 产物的衔接方式。
- 设计实体层级、讨论强度和跨平台确认的表现语义。
- 设计外部层缺失时的降级表达与审计口径。
