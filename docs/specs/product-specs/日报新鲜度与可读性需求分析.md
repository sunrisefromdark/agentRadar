# 日报新鲜度与可读性需求分析

## 文档状态

- 状态：`Frozen for Design`
- 需求级别：`S2 — Medium Feature`
- 分析方法：依据 [需求分析_skill.md](需求分析_skill.md#L1) 输出
- 本文目标：在不讨论实现方案的前提下，冻结“日报新鲜度可见性、候选池统计语义、日报可读性增强”的产品需求

## 1. 输入研读结论

### 目标

- 让用户先判断“这份日报是否足够新鲜、是否值得继续看”
- 让 daily report 优先聚焦“当天的新鲜明星项目”，而不是把历史 fallback 内容和当天内容混在同一优先级
- 让当天明星项目的发现层以实时拉取为前提，而不是依赖历史 digest 回放
- 让用户正确理解 `候选项目总数`、`今日新增`、`单日出现项目` 一类统计的真实含义
- 让用户快速看懂重点项目“它是什么、为什么今天值得看”

### 用户

- 每日查看趋势雷达结果的分析者
- 先关心数据是否可靠，再决定是否相信榜单的决策者
- 需要快速浏览重点项目而不是逐行读指标的产品/技术用户

### 当前痛点

- `fallback` 发生时没有被放在最显眼位置，用户可能先读内容，后意识到数据并不新鲜
- `候选项目总数 81/82`、`单日出现项目 80` 之类数字容易被误解为系统固定阈值或真实趋势结论
- 发现层当前会被历史 digest / snapshot 回退污染，导致 daily report 名义上是“今天”，实质上却在复用旧候选
- daily report 更像“调试输出来历堆叠”，不适合快速阅读
- 项目缺少“人话简介”，用户需要自己从 tags、分数和证据里倒推项目是什么

### 触发条件

- 用户跑完 `run-daily` 后第一时间打开日报
- 用户比较多天日报时，发现候选规模长期稳定，怀疑是否存在固定上限或历史污染
- 用户看到大量 `single-source` / `single-spike` 提示，但不清楚那是观测口径还是项目本身特征

### 已知约束

- 当前系统核心评分链路是 `rules-only`
- LLM 不是主判定依赖，只允许作为可选 semantic classification 辅助
- `agents-radar` 不是 GitHub 官方接口，而是本地接入的上游 digest 系统；它可以作为上下文来源，但不应再主导 daily 主榜单发现层
- 产品核心追求之一是“时效性与消息新鲜性”
- daily report 的主价值是帮助用户判断“当天值得看的明星项目”，不是做历史归档浏览

## 2. 需求规模判断

本需求判定为 `S2 — Medium Feature`。

原因：

- 涉及多模块协同：run summary、daily report、统计口径、用户理解路径
- 会明显改变用户的阅读顺序和决策方式
- 不只是文案调整，而是“信息优先级”和“统计语义”的产品口径重构

## 3. 多义解读

### 诉求 A：为什么每天候选总数都在 `81/82`

可能含义 1：
用户怀疑系统存在隐藏上限、固定截断或榜单只取 Top-K。

可能含义 2：
用户怀疑当前候选池其实是由固定历史源反复回放产生，不能代表“当天的新鲜趋势覆盖”。

本次需求按“含义 2 为主，含义 1 为辅”处理，因为用户核心诉求是新鲜度与可信度，而不是单纯排查 hard cap。

### 诉求 B：`单日出现项目` 这个口径不清楚

可能含义 1：
项目真的只在真实世界中火了一天。

可能含义 2：
系统当前只对该项目采到一次有效观测，尚未形成跨天或跨源确认。

本次需求冻结为“含义 2 必须优先表达”，禁止再使用容易误导成真实趋势结论的模糊词。

### 诉求 C：日报太丑

可能含义 1：
排版不好看。

可能含义 2：
信息层级错误，用户先看到一堆术语和指标，无法先判断这份报告是否值得读、重点项目是什么。

本次需求按“含义 2”为准，重点解决信息结构，不把它降格成纯视觉排版问题。

## 4. 用户与场景建模

### 用户角色

- 数据可靠性优先型用户：先看 freshness，再决定是否看内容
- 趋势扫描型用户：想快速知道今天哪些项目值得关注
- 解释核对型用户：想理解为什么一个项目上榜，但不希望第一屏就被技术证据淹没

### 使用场景

#### 场景 1：早晨例行查看日报

- 触发：用户每日打开最新 `daily report`
- 目标：先确认数据是不是新鲜，再决定是否继续阅读当天明星项目

#### 场景 2：发现 fallback 后判断是否可用

- 触发：某个 source 回退到几天前
- 目标：用户立刻意识到报告存在时效性下降，并判断这次报告还能不能作为决策依据

#### 场景 3：快速浏览重点项目

- 触发：用户没有时间看完整证据串
- 目标：在前几屏内知道“项目是干什么的、为什么今天值得看”

### 关键路径

1. 用户打开日报
2. 首屏先看到总状态结论、数据新鲜度与候选池健康情况
3. 用户确认数据可读后，进入“当天明星项目”列表
4. 用户对单个项目先读人话简介，再按需查看详细证据

## 5. 产品定义

### 需求 1：新鲜度优先展示

- What：日报首屏必须优先展示当前数据新鲜度、fallback 情况、候选池可靠性
- Why：如果实时拉取失败并回退到历史快照，用户需要在最早阶段知道，而且系统不能再把这些旧候选伪装成当天发现
- Who：所有日报阅读者
- When：打开 daily report 的第一屏
- Expected outcome：用户能在 5 秒内判断“这份日报值不值得继续看”

### 需求 1B：发现层实时性

- What：daily report 的当天明星项目发现层必须优先来自当次执行时的实时拉取结果
- Why：daily report 的核心价值是“今天有什么新鲜热点”，不是复述几天前的 digest
- Who：所有日报阅读者
- When：运行 `run-daily` 并生成主榜单时
- Expected outcome：当日主榜单以 GitHub / Trendshift 的当次实时结果为基础；实时拉取失败时，主榜单显式降级而不是继续吃旧候选

### 需求 1A：首屏总状态结论

- What：日报首屏必须给出一个总状态结论，直接概括当前数据是否足够新鲜、是否适合用于判断当天热点
- Why：用户不应该自己从多行 source 状态里拼凑出“今天这份日报到底可不可信”
- Who：所有日报阅读者
- When：用户打开日报的第一眼
- Expected outcome：用户无需阅读正文即可知道“可直接阅读 / 谨慎参考 / 不建议据此判断”

### 需求 2：统计口径去歧义

- What：把当前容易误导的统计字段改成明确、可解释、无歧义的中文口径
- Why：避免用户把“观测覆盖结果”误读成“市场真实趋势结论”
- Who：所有日报阅读者，尤其是第一次接触系统的人
- When：阅读 summary 和 daily report 顶部统计时
- Expected outcome：用户不再需要借助代码或口头解释理解这些数字

### 需求 3：项目卡片人话化

- What：重点项目必须提供两条中文摘要：
  - 一句话说明“项目是做什么的”
  - 一句话说明“为什么今天值得关注”
- Why：降低阅读门槛，让用户先获得项目认知，再决定是否深入证据细节
- Who：趋势扫描型与决策型用户
- When：阅读 Top 项目或观察列表时
- Expected outcome：用户在不看术语和分数组件的前提下，也能快速理解上榜原因

### 需求 4：详细证据下沉

- What：将复杂指标、术语和分数组件下沉到详细层，不再占据第一阅读层
- Why：日报给用户看的第一层不应是调试型输出
- Who：所有日报阅读者
- When：浏览重点项目时
- Expected outcome：第一层内容更像“结论摘要”，第二层才是“审计证据”

## 6. 业务规则与约束

### 数据新鲜度规则

- 任何 `fallback` 都必须视为用户可感知的重要状态，而不是仅写在长段 notes 里
- 当存在回退、历史快照、过期 source 时，日报首屏必须显式提示
- 产品层必须把“数据不够新鲜”视为重要风险，而不是普通说明文字
- 当日报中的核心候选主要来自非当天数据时，系统不得让用户误以为这是“今日新鲜热点”
- 对 daily 主榜单而言，实时发现失败不是普通降级，而是核心能力缺失

### 实时发现层规则

- daily 主榜单的候选发现必须优先来自命令执行当次的实时拉取结果
- 以下来源属于 daily 主榜单的实时发现层：
  - GitHub Trending
  - GitHub 当次拉取后识别出的显著 star 日增项目
  - watchlist orgs 中当次拉取后存在新鲜 GitHub 动态的项目
  - Trendshift 当次拉取得到的当天推荐项目
- `agents-radar`、历史 snapshot、历史 digest 只能作为上下文或补充观察来源
- 当实时发现层失败时，系统可以保留历史内容用于“补充观察”，但不得继续用它们生成“当天明星项目”主榜单

### 当天明星项目规则

- daily report 的主列表必须优先代表“当天值得看的明星项目”
- “当天明星项目”优先来源于以下当日新鲜信号：
  - GitHub Trending
  - watchlist orgs 中当天有明显新动态的项目
  - star 日增量显著且满足当前筛选要求的项目
  - Trendshift 当天最新推荐且符合要求的项目
- 历史 fallback 内容可以作为补充上下文存在，但不得与当天新鲜项目以同等优先级混排
- 当系统无法形成足够的当日新鲜候选时，日报必须显式告知“今天主榜单并不新鲜”
- `agents-radar` 不得再作为 daily 主榜单的一线发现源；它只能作为辅助解释、长期跟踪或补充观察来源

### 统计口径规则

- 禁止继续使用会让用户误判为“真实市场趋势”的模糊口径
- “单日出现项目”必须改成更明确表达，明确指向“观测层含义”而不是“市场层含义”
- `候选项目总数` 必须能被解释为系统当前观测范围，而不是暗示“全网真实规模”
- 用户展示层中，`单日出现项目` 的替代口径冻结为：`待二次确认项目`
- `待二次确认项目` 的定义必须固定为：当前仅获得一次有效观测，尚未形成跨天或跨源确认

### 内容表达规则

- 第一屏优先回答“今天整体数据质量和候选池情况如何”
- 第一屏必须给出总状态结论
- 中文项目简短介绍必须偏“人话定义 + 今日关注理由”
- 给用户看的日报不得以大量评分组件和术语作为首层内容

### 判断链路规则

- 当前关键判断链路的产品定义仍以规则评分为主
- LLM 不得被定义成日报可信性的前置依赖
- 如果未来启用 LLM，也只能做辅助分类，不能替代主判断口径
- 当前产品定义下，LLM 不是“今日是否上榜”的核心判断源；主判断标准仍应保持可解释、可审计的规则判定

### watchlist 与用户偏好规则

- 当前 `watchlist_orgs` 的产品语义固定为：组织级重点监控名单
- `watchlist_orgs` 可以手动配置，但它表达的是“哪些组织值得优先监控”，不是“这些项目天然更重要”
- 因此，`watchlist_orgs` 命中可以提升：
  - 发现优先级
  - 展示优先级
  - 人工审查优先级
- `watchlist_orgs` 命中本身不得直接提升客观趋势分

- 若要支持“用户当前更关注什么方向”，必须新增独立的用户偏好层，而不是复用 `watchlist_orgs`
- 本轮范围内，用户偏好层采用最小版本落地：来源先放在配置中，由用户手动维护
- 当前配置入口固定为 `config.yaml`
- 用户偏好层表达的是：
  - 更关注 `agent runtime`
  - 更关注 `memory`
  - 更关注 `coding agent`
  - 更关注 `infra / evaluation / MCP`
- 这类“用户主动声明的关注方向”可以进入加权体系，但必须满足：
  - 只作为单独的偏好加权项
  - 不与客观趋势分混成黑箱总分
  - 日报展示层只输出一个最终排序，不额外展示多套榜单
- 前端页面输入偏好不属于本轮范围，后续等前端能力落地后再承接

## 7. 去歧义后的需求口径

以下表达在设计阶段前必须统一：

- `单日出现项目`
  改为：
  `待二次确认项目`
  并固定定义为：
  `当前仅获得一次有效观测，尚未形成跨天或跨源确认`

- `日报优化`
  明确为：
  `重构信息层级，让首屏优先展示数据新鲜度与候选池可靠性，并为重点项目提供人话简介`

- `fallback 提示`
  明确为：
  `当任一核心 source 不是当天新鲜数据时，用户必须在报告最显眼位置感知到`

- `daily report`
  明确为：
  `优先呈现当天新鲜候选中的明星项目，而不是默认混入历史 fallback 项目`

- `实时发现层`
  明确为：
  `通过当次执行实时拉取 GitHub / Trendshift 得到的候选发现层，而不是依赖历史 digest / snapshot 回放`

- `watchlist_orgs`
  明确为：
  `组织级重点监控名单，只影响发现/展示优先级，不直接提升客观趋势分`

- `用户偏好层`
  明确为：
  `用户主动声明的关注方向；本轮先通过配置输入，参与单一最终排序的偏好加权，但不得污染客观趋势分`

## 8. 验收标准

### 需求 1：新鲜度优先展示

- 可观测结果：
  打开日报第一页即可看到是否发生 fallback、哪些 source 不是当天数据、当前内容是否存在时效性风险
- 成功条件：
  用户不需要读到正文中部或 run summary 才发现数据是旧的
- 失败条件：
  fallback 信息仍然埋在说明段落、notes 或附录里

### 需求 1B：发现层实时性

- 可观测结果：
  daily 主榜单的候选来源可被说明为“当次实时拉取得到”，而不是来自旧 digest 回放
- 成功条件：
  当 GitHub / Trendshift 实时拉取正常时，主榜单以这些实时结果为主；当实时拉取失败时，主榜单显式降级或为空，而不是被历史候选自动填满
- 失败条件：
  报告仍然主要依赖几天前的 digest / snapshot 生成当天主榜单

### 需求 1A：首屏总状态结论

- 可观测结果：
  日报首屏存在总状态结论，直接说明本次报告是否适合用于判断当天热点
- 成功条件：
  用户第一眼即可理解当前属于：
  - 数据新鲜，可直接阅读
  - 数据部分回退，谨慎参考
  - 数据显著过期，不建议直接用于判断
- 失败条件：
  用户仍需自己从多条 source 说明推断整体状态

### 需求 2：统计口径去歧义

- 可观测结果：
  候选池相关数字都带有明确中文定义，用户可直接理解其含义
- 成功条件：
  用户不会再把 `81/82` 理解为隐藏上限，也不会把“单次观测”误解为真实世界只热一天
- 失败条件：
  数字仍需借助源码、开发口头解释或历史背景才能理解

### 需求 3：项目卡片人话化

- 可观测结果：
  每个重点项目都至少有“项目是做什么的”和“今天为什么值得关注”两条中文摘要
- 成功条件：
  用户可以先靠摘要理解项目，再决定要不要读详细证据
- 失败条件：
  项目仍主要以标签、分数、组件项和术语堆叠呈现

### 需求 4：详细证据下沉

- 可观测结果：
  指标组件和审计细节不再占据项目第一展示层
- 成功条件：
  第一层内容以结论和用户可读解释为主
- 失败条件：
  daily report 仍然像工程调试日志

### 需求 5：当天明星项目优先

- 可观测结果：
  daily report 主列表优先体现当日新鲜来源里的明星项目
- 成功条件：
  用户打开日报后，看到的是当天热点候选，而不是被历史 fallback 项目主导
- 失败条件：
  报告虽然名为 daily，但主榜单主要来自几天前的回退源，且未明确降级

### 需求 6：用户偏好与客观趋势分离

- 可观测结果：
  系统能够区分“组织 watchlist 命中”和“用户兴趣偏好”两类信号
- 成功条件：
  `watchlist_orgs` 不直接改写客观趋势分；用户偏好通过配置输入生效，日报展示层只保留一个最终排序，但审计层仍可区分客观分与偏好增益
- 失败条件：
  组织 watchlist 和用户兴趣偏好被混用，或用户偏好直接污染唯一总分且无法分离审计

## 9. 风险与缺口

### 风险

- 如果不先定义“新鲜度优先”，后续再做视觉优化也只会把旧问题包装得更好看
- 如果不澄清统计口径，用户会继续误读系统输出，导致错误决策
- 如果把项目简介写成术语摘要或营销文案，会损害日报可信度

### 缺口

- 当前“多旧才算不新鲜”的 source 分级阈值仍需在设计阶段写成明确状态映射
- 当前“当天明星项目”在多来源同时存在时的优先排序规则尚未冻结到最终展示级别

## 10. 需求冻结检查

当前已经冻结的内容：

- 接受废弃“单日出现项目”这一模糊说法
- 用户展示层的新口径固定为：`待二次确认项目`
- 项目简介应采用“项目是做什么的 + 为什么今天值得关注”的双句结构
- daily report 首屏必须优先回答“今天整体数据质量和候选池情况如何”
- daily report 首屏必须提供总状态结论
- fallback 必须在最显眼位置提示
- 产品核心追求之一是新鲜性，因此“旧数据却不显著提示”属于核心体验缺陷
- daily report 的主目标是呈现当天明星项目；GitHub Trending、watchlist 当日动态、star 日增显著项目、Trendshift 当日推荐应被视为优先新鲜来源
- `agents-radar` 应按“上游 digest 来源”来定义，而不是误写成 GitHub 官方直连源；它不是 daily 主榜单的一线实时发现源
- 实时发现层必须由 GitHub / Trendshift 当次拉取得到，不能再用历史 digest / snapshot 默认充当 daily 主榜单来源
- 当前关键判断标准应按“规则评分主导、LLM 非必需辅助”来定义
- `watchlist_orgs` 固定为组织级重点监控名单，不直接提升客观趋势分
- 用户偏好层属于本轮范围，但先采用“配置输入、手动维护”的最小版本
- 前端页面输入偏好不属于本轮范围，后续应由 UI 相关计划承接
- 用户偏好层必须与客观趋势分保持分离
- 日报展示层只保留一个最终排序，不展示多套排序模式

仍未冻结、会阻塞后续设计的问题：

1. “数据新鲜，可直接阅读 / 数据部分回退，谨慎参考 / 数据显著过期，不建议直接用于判断” 这三档状态如何映射到具体 source 组合，仍需在设计阶段量化
2. “当天明星项目”在 GitHub 新鲜源与 Trendshift 新鲜源同时出现时，是否要按来源类型再做固定优先序，仍需设计阶段冻结
