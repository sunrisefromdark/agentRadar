# 自然语言模糊搜索与推荐需求分析

## 文档状态
- 状态：`READY`
- 需求级别：`S2 - Medium Feature`
- 分析方法：依据 [需求分析_skill.md](需求分析_skill.md#L1) 输出
- 本文目标：在不讨论实现细节的前提下，冻结一版“精确搜索之外的自然语言模糊意图推荐入口”需求边界，用于直接进入设计阶段

## Requirement Size
`S2 - Medium Feature`

判定原因：
- 本需求不是单点文案修改，而是新增一层用户可感知的查询语义：从“按名称/方向检索”扩展到“按自然语言意图请求推荐”。
- 它会同时影响查询入口、结果组织、结果解释、失败语义与产品边界表达。
- 它不改写底层 signal / scoring / weekly judgment 主链路，也不要求引入新的外部数据源，因此未达到 `S3` 的系统级重构规模。

## 1. Background

### 需求来源
- 当前项目搜索更接近“显式词命中”能力，主要适合公司名、方向名、项目名、repo 名等已知目标查询。
- 当用户提出“我想看看最热的股票有什么”“最近有什么值得追”“有没有偏低风险但热度高的项目”这类语义模糊请求时，系统缺少正式定义的推荐式响应能力。
- 用户希望系统不仅能“找名字”，还能“理解我想看哪类东西”，并基于现有产物给出可解释推荐。

### 业务背景
- 本项目当前核心真相层已经具备 `daily / weekly / observer / kb / run health` 等可消费 artifact。
- 当前系统已经产出 `project_brief_cn`、`why_today_cn`、`paradigm`、`matched_interest_topics`、`risk`、`persistence` 等语义字段，但这些字段主要用于展示和轻量个性化，不等于正式的自然语言推荐入口。
- 产品主价值仍然是“AI Agent 生态内的可解释趋势判断”，不是通用搜索引擎，也不是开放域问答系统。

### 用户动机
- 用户往往先有“想看什么类型的东西”的意图，再有明确项目名。
- 用户不一定知道应搜索 `MCP`、`runtime`、`memory`、`coding agent` 这些内部标签词，但仍然希望系统能把意图翻译成结果。
- 用户希望看到“系统是怎么理解我的请求的”以及“为什么这些结果被推荐出来”。

## 2. Goals

### 产品目标
- 在保留现有精确搜索能力的前提下，新增一层“自然语言模糊意图 -> 可解释推荐结果”的产品能力。
- 让系统支持“推荐式查询”而不退化为黑盒聊天回答。
- 保持所有推荐结果都绑定到现有 artifact、已有项目对象或已有趋势对象，不引入第二套真相源。
- 让系统对模糊请求给出受控解释、受控推荐和受控失败，而不是直接无结果或胡乱泛化。

### 用户价值
- 用户即使不知道精确关键词，也能获得一组“此刻值得看”的候选结果。
- 用户可以快速理解系统把请求解释成了什么，以及结果为什么匹配。
- 用户可以区分“系统没搜到精确词”和“系统理解了意图但当前没有足够结果”。

## 3. Users & Scenarios

### 用户角色
- 趋势扫描型用户：想快速知道“最近有什么值得看”。
- 半熟悉生态的用户：知道自己想看一类东西，但不知道精确术语或项目名。
- 深度判断型用户：希望按“热度 / 风险 / 持续性 / 趋势形成度”这类组合条件找候选。
- 回访型用户：想用更自然的话复查“上周说的那类方向现在还有没有新东西”。

### 使用场景

#### 场景 1：用户不知道项目名，只知道想看“最热的”
- 触发：用户输入“最近最热的都有什么”“我想看看现在最火的 agent 项目”。
- 目标：系统返回当前时间片里最值得优先浏览的一组候选，并说明默认采用的热度语义。

#### 场景 2：用户使用模糊口语或比喻性说法
- 触发：用户输入“我想看看最热的股票有什么”“最近有什么龙头项目”“给我看看最值得抄作业的方向”。
- 目标：系统把口语或比喻翻译成站内可支持的推荐语义，并显式告知解释方式。

#### 场景 3：用户按组合条件找结果
- 触发：用户输入“有没有最近很热但风险没那么高的”“最近一周形成趋势的 MCP 项目有哪些”。
- 目标：系统能同时理解主题、时间窗口、热度、风险、持续性等意图维度。

#### 场景 4：用户其实问的是站外领域
- 触发：用户输入“美股里最热的股票是什么”“A 股最近最强板块有哪些”。
- 目标：系统不伪装成金融搜索，而是清楚表达本产品只覆盖 AI Agent 生态项目，不提供真实证券推荐。

### 触发条件
- 用户没有明确项目名、公司名或 repo 名。
- 用户请求的是“推荐”“看看有什么”“最热 / 值得追 / 风险低 / 新方向”等组合意图。
- 用户使用了口语化、模糊化或比喻化表达。

### 关键用户路径
1. 用户输入自然语言模糊请求。
2. 系统先给出“按以下方式理解你的请求”的解释。
3. 系统基于当前时间片和既有 artifact 返回受控推荐结果。
4. 用户继续点进项目、趋势或 KB 详情，完成后续判断。

## 4. Requirement Definition

### 需求 1：系统必须把“推荐式自然语言查询”定义为独立于精确搜索的正式能力
- What：产品必须显式支持“按意图请求推荐”的查询模式，而不把它伪装成失败后的关键词搜索补丁。
- Why：用户需求不是“名字拼错了”，而是“根本不知道该搜什么词”。
- Who：趋势扫描型用户、半熟悉生态的用户、深度判断型用户。
- When：用户输入推荐式、模糊式、口语式请求时。
- Expected outcome：用户能感觉到系统在做“理解意图并推荐”，而不是“只有关键词命中才算成功”。

### 需求 2：系统必须显式输出“请求解释”
- What：每次推荐式查询都必须告诉用户系统如何解释了本次请求，至少包含对象范围、时间语义、主要排序意图或过滤意图。
- Why：模糊查询天然存在多义，如果不先解释，用户无法判断结果是“没理解”还是“理解错了”。
- Who：所有使用推荐式查询的用户。
- When：推荐结果展示前。
- Expected outcome：用户能看到类似“已按『当前时间片内的热门项目推荐』理解”“已按『AI Agent 生态中的热门标的』理解”的解释。

### 需求 3：系统必须支持受控的模糊意图维度，而不是无限开放解释
- What：V1 必须至少支持以下意图维度的受控组合：
  - 关注对象：项目 / 趋势方向
  - 时间语义：今天 / 最近 / 本周 / 当前可用结果
  - 热度意图：最热 / 值得看 / 值得继续跟
  - 风险与可信度意图：高置信 / 风险较低 / 仍需谨慎
  - 持续性意图：新冒头 / 持续出现 / 已形成趋势
- Why：用户的模糊请求通常不是单一关键词，而是“对象 + 时间 + 判断维度”的组合。
- Who：趋势扫描型用户、深度判断型用户。
- When：用户输入复合自然语言请求时。
- Expected outcome：系统能在站内已有语义维度内做受控理解，而不是把任意自然语言都当作开放问答。

### 需求 4：系统必须支持“口语 / 比喻词”到站内语义的有限映射
- What：系统必须允许一小类近域口语或比喻词映射到站内对象语义，例如“股票 / 标的 / 龙头 / 值得抄作业的项目”可被解释为“值得优先关注的项目候选”，但该映射必须受控且可见。
- Why：真实用户经常不会使用系统内部的正式名词。
- Who：半熟悉生态的用户、趋势扫描型用户。
- When：用户请求明显仍然围绕本产品领域，但用词带有口语或比喻色彩时。
- Expected outcome：用户能得到结果，并看到“系统按站内语义做了怎样的翻译”。

### 需求 5：系统必须对“真正站外”的请求给出边界响应
- What：当用户请求明显超出 AI Agent 生态范围，例如真实证券、财报、股价、A 股 / 美股 / 加密资产行情时，系统必须拒绝伪造推荐，并清楚表达产品边界。
- Why：本产品的真相层不覆盖真实金融资产，错误承接会制造高风险伪结果。
- Who：所有用户。
- When：请求涉及明确站外领域时。
- Expected outcome：用户看到“本产品只覆盖 AI Agent 生态项目，不提供真实股票推荐；如果你想看站内热门项目，我可以继续按项目标的推荐”的受控边界表达。

### 需求 6：推荐结果必须绑定现有 artifact 语义，不得制造第二套排序真相
- What：V1 一等推荐结果只能来自当前已存在的 `daily projects` 与 `weekly trends`；`observer` 与 `KB` 只能作为辅助证据、解释支撑、方向孵化来源或下钻详情出现。推荐层不得重新发明新的总分、热度值或趋势事实，也不得把 `observer / KB` 条目提升为一等推荐结果。
- Why：本项目核心价值在于 artifact-first 与可审计；推荐入口不能把产品重新变成黑盒。
- Who：所有用户，以及后续设计 / 实现人员。
- When：定义结果组织、结果排序与解释时。
- Expected outcome：推荐结果只是“对现有对象的查询与组织方式”，而不是“新的事实计算层”。

### 需求 7：系统必须把推荐结果的对象范围表达清楚
- What：系统必须明确区分推荐结果是“项目级结果”还是“趋势级结果”；若同一查询同时返回两类结果，也必须分组呈现，不能混在一起伪装成同一类对象。
- Why：项目和趋势的判断语义不同，用户不能在一个无标签列表里混读。
- Who：所有使用推荐式查询的用户。
- When：查询触发多种对象范围时。
- Expected outcome：用户能明确知道“这是项目推荐”“这是本周趋势方向推荐”，而不是只看到一串混合名词。

### 需求 8：推荐结果必须带“为什么匹配这次请求”的解释
- What：每个被推荐结果必须至少有一条与本次查询相关的匹配理由，例如“命中 MCP 方向”“当前热度高”“本周持续出现”“风险提示较少”“属于弱信号待观察”。
- Why：推荐入口的可信度不只来自结果，还来自“为什么是它”。
- Who：所有用户。
- When：展示推荐结果时。
- Expected outcome：用户不需要反向猜测系统为什么选了这些对象。

### 需求 9：系统必须对“无结果 / 低置信解释 / 需澄清”三类失败语义做区分
- What：推荐式查询失败时，至少要区分：
  - 无匹配结果：系统理解了请求，但当前 artifact 中没有满足条件的对象
  - 解释不稳定：系统发现请求有多种高冲突解释，当前不能自信给出单一推荐
  - 超出产品范围：请求不属于本产品覆盖领域
- Why：这些失败不是一回事，混成一个“没搜到”会损害可用性。
- Who：所有用户。
- When：查询无法稳定返回结果时。
- Expected outcome：用户知道是“没有结果”“理解不稳”还是“超域请求”，并获得下一步建议。

### 需求 10：该能力必须在 `rules-only` 模式下仍可降级可用
- What：即使未启用 LLM，系统也必须对一部分常见模糊意图提供可用的规则型推荐；若启用 LLM，LLM 也只能做查询解释增强，不能越权生成最终事实。
- Why：仓库已有明确约束：`rules-only` 必须可运行，LLM 不能成为唯一真相入口。
- Who：所有用户，尤其是本地只读或未配 key 的使用者。
- When：LLM 未启用、调用失败或系统运行在 rules-only 时。
- Expected outcome：系统不会因为没有 LLM 就完全丧失模糊推荐能力，只会在覆盖面或表达细腻度上降级。

## 5. Interpretation Options

### 关键请求 A：“做语义搜索”
- 解释 A1：把现有搜索改成关键词扩展、同义词扩展和模糊匹配。
- 解释 A2：新增一层“自然语言意图推荐”，允许用户直接请求“给我看一类值得看的对象”。
- 推荐冻结：采用 `A2`，但保留精确搜索作为并行能力。
- 原因：用户问题的本质不是拼写模糊，而是意图表达先于目标名称。

### 关键请求 B：“我想看看最热的股票有什么”
- 解释 B1：按真实金融证券语义处理，返回股票市场结果。
- 解释 B2：在本产品领域内，把“股票 / 标的”视为“值得优先关注的项目候选”的口语说法。
- 推荐冻结：采用 `B2` 作为近域口语映射，但前提是整体上下文仍然指向 AI Agent 生态；若用户明确提到 A 股 / 美股 / 股价，则回退到超域边界响应。
- 原因：这类表达在站内更像比喻性浏览请求，而不是金融数据请求。

### 关键请求 C：“推荐结果怎么排”
- 解释 C1：推荐层重新计算一套黑盒分数，再按新分数排序。
- 解释 C2：推荐层只在既有 artifact 对象上做查询解释、候选收敛、过滤与组织，保留原始对象事实和原始排序依据可追溯。
- 推荐冻结：采用 `C2`。
- 原因：仓库现有产品原则禁止新入口私自制造第二套事实排序。

### 关键请求 D：“遇到歧义怎么办”
- 解释 D1：一旦存在歧义就强制用户先确认，没确认不给结果。
- 解释 D2：对常见模糊请求优先做 best-effort 解释并显式展示解释结果；仅在冲突过大时才进入澄清或不给结果。
- 推荐冻结：采用 `D2`。
- 原因：若所有模糊请求都要求二次确认，推荐入口会退化成高摩擦检索。

## 6. Constraints

### 权限与边界
- 本能力服务于本项目的 AI Agent 生态对象，不扩展到真实证券、行情、财报或其他站外资产。
- 本能力不改变现有 exact search、项目详情、weekly、observer、KB 的对象所有权。
- 本能力第一版不要求用户先登录才能使用；若未来接入登录态偏好，也不能改变匿名下的基础可用性。

### 数据归属与真相层边界
- 推荐层只能消费现有 artifact 与其字段，不得直接引入新的实时外部搜索结果作为正式推荐真相。
- 推荐层不得重算 `score / confidence / trend judgment / observer score` 的业务语义。
- 推荐解释必须可追溯到现有字段，如 `project_brief_cn`、`why_today_cn`、`paradigm`、`risk`、`persistence`、`weekly core trend`、`weak signal` 等。

### 业务规则约束
- 精确搜索与推荐式查询必须共存，不能互相覆盖。
- 推荐结果必须带对象标签与时间上下文，不能脱离 `date / weekly window / latest` 语义单独存在。
- 如果查询解释依赖 LLM，最终展示仍必须符合现有 artifact 边界与 `rules-only` 底线。
- 推荐层可以帮助用户“更快找到对象”，但不能把“用户兴趣”伪装成“客观热度”。
- V1 的一等推荐结果对象冻结为 `daily projects + weekly trends`；`observer` 仅作为辅助证据或方向孵化来源，`KB` 仅作为下钻详情与解释支撑，不直接作为一等推荐结果对象。
- V1 的近域口语 / 比喻映射必须先冻结为有限白名单；白名单外表达不得自由扩张为站内语义，只能进入澄清、失败或保守降级语义。
- 当同一请求同时存在“项目推荐”和“趋势推荐”两种可行解释时，V1 默认输出一个主解释，并可附带次解释；只有在高冲突且无法稳定判断时才进入“需澄清”。

### 兼容性约束
- 第一版必须兼容 `visual-console` 与 `visual-console:web` 的只读消费模式。
- 在 `llm.enabled=false` 时，常见模糊请求仍需有可用降级行为。
- 在当前 artifact 不足、weekly 不可判、observer 缺失时，推荐入口也必须沿用既有失败与降级语义。

## 7. Acceptance Criteria

### 验收 1：常见模糊热度请求可返回推荐
- 可观察结果：用户输入“最近最热的都有什么”后，系统返回一组站内对象，并先显示请求解释。
- 成功条件：用户无需知道项目名，也能拿到一组当前可浏览候选。
- 失败条件：系统直接返回空白、仅要求输入精确名称，或输出与请求无关的随机对象。

### 验收 2：近域口语 / 比喻请求可被受控翻译
- 可观察结果：用户输入“我想看看最热的股票有什么”时，系统明确显示已按“热门项目标的”理解，并返回站内推荐。
- 成功条件：结果和解释都保持在 AI Agent 生态边界内。
- 失败条件：系统把它误答成真实证券推荐，或完全不处理口语表达。

### 验收 3：超域金融请求被正确拦截
- 可观察结果：用户输入“美股最近最热的股票是什么”时，系统明确说明不覆盖真实股票市场。
- 成功条件：系统不伪装有能力回答站外金融问题，并给出站内可转向建议。
- 失败条件：系统输出虚假金融推荐，或把超域请求默默映射成站内结果而不提示。

### 验收 4：结果解释清楚说明系统如何理解了请求
- 可观察结果：结果页存在一段简短解释，至少包含对象范围、时间语义和主要推荐标准。
- 成功条件：用户能判断“结果是不是按我想要的那种方式在推荐”。
- 失败条件：系统只给结果，不给解释，导致用户无法判断是否理解正确。

### 验收 5：每个结果都有与本次请求相关的匹配理由
- 可观察结果：推荐结果带有“为什么匹配这次请求”的短解释。
- 成功条件：用户能区分“因为热度高被推荐”“因为本周持续出现被推荐”“因为命中 MCP 方向被推荐”等不同理由。
- 失败条件：推荐结果只有项目名和总分，没有本次查询相关的匹配说明。

### 验收 6：对象范围不混淆
- 可观察结果：项目级结果、趋势级结果若同时出现，会被显式分组或显式标注。
- 成功条件：用户知道自己当前看到的是项目推荐还是趋势方向推荐。
- 失败条件：不同对象被混成一个列表，用户无法理解阅读口径。

### 验收 7：失败语义可区分
- 可观察结果：系统能区分“无结果”“解释不稳定”“超出产品范围”。
- 成功条件：不同失败原因对应不同提示和下一步建议。
- 失败条件：所有失败都显示为“没有搜索结果”。

### 验收 8：`rules-only` 下仍可基础可用
- 可观察结果：在未启用 LLM 时，至少“最热的”“最近值得看”“MCP 方向里有什么”等常见请求仍有可用响应。
- 成功条件：模糊推荐能力降级但不消失。
- 失败条件：没有 LLM 时推荐式查询整体不可用。

### 验收 9：一等推荐结果对象边界清晰且不越权
- 可观察结果：推荐主结果只出现 `daily projects` 与 `weekly trends` 两类对象；`observer` 或 `KB` 若被引用，只以辅助证据、解释支撑或下钻信息出现。
- 成功条件：用户能清楚区分主推荐结果与辅助证据层，不会把 `observer` 或 `KB` 误读为与项目 / 趋势并列的一等推荐对象。
- 失败条件：系统把 `observer` 命中或 `KB` 条目直接混入主推荐结果，伪装成正式推荐对象。

### 验收 10：近域口语映射只在冻结白名单内生效
- 可观察结果：用户输入白名单内口语表达时，系统能按站内语义做受控翻译；输入白名单外近义表达时，系统会进入澄清、失败或保守降级，而不是继续自由扩张解释。
- 成功条件：常见近域口语可被稳定承接，同时产品边界不会因自由扩张而失控。
- 失败条件：系统把未冻结的口语 / 比喻词也默认映射为站内语义，导致超域请求或低相关请求被误吸纳。

### 验收 11：双重解释冲突时采用主解释优先，必要时才澄清
- 可观察结果：当同一请求同时可落到“项目推荐”和“趋势推荐”时，系统默认先给出一个主解释，并可附带次解释；仅在高冲突场景下输出“需澄清”。
- 成功条件：用户大多数情况下能直接获得可消费结果，同时仍能看见系统如何做了解释取舍。
- 失败条件：系统对轻度歧义也一律要求用户先确认，或在高冲突场景下仍强行给出不稳定推荐。

## 8. Risks & Gaps

### 风险
- 如果口语 / 比喻映射范围过宽，系统会把超域请求误吸纳为站内推荐。
- 如果推荐层重新制造一套排序分数，用户会混淆“推荐顺序”和“客观榜单顺序”。
- 如果项目级与趋势级结果混排，用户会失去结果语义。
- 如果 LLM 解释越权，系统可能在 `rules-only` 外形成不可审计的黑盒行为。
- 如果失败语义不细分，用户会把“产品边界问题”误认为“搜索质量差”。

### 当前缺口
- 设计阶段必须把“V1 首批支持的查询意图词典”和“近域口语映射白名单”沉淀成可审计清单；未形成清单前不得进入 ExecPlan。产品边界已冻结为“有限白名单映射，不自由扩张”。
- 设计阶段必须把“高冲突解释必须要求澄清”的判断门槛量化为可复用规则口径；未形成规则前不得进入 ExecPlan。产品默认策略已冻结为“主解释优先，仅高冲突时澄清”。

### 隐含假设
- 用户愿意接受“项目推荐”和“趋势推荐”是两类不同但都成立的推荐语义，只要系统显式说明当前采用了哪一种主解释。
- 用户愿意接受近域口语的受控翻译，但不期待系统无限扩张到真实金融、开放域问答或站外资产检索。
- V1 阶段用户更在意“即使不知道精确关键词也能拿到可信候选”，而不是立即获得基于长期画像的个性化排序。

## 9. Open Questions

以下问题已结合当前产品判断完成冻结，可作为设计阶段输入：

1. V1 的一等推荐结果对象是否只覆盖 `daily projects + weekly trends`，还是同时覆盖 `observer + KB`？
   已冻结：V1 的一等推荐结果对象只覆盖 `daily projects + weekly trends`；`observer` 只作为辅助证据与方向孵化来源，`KB` 只作为下钻详情与解释支撑，不直接进入一等推荐结果。

2. “股票 / 标的 / 龙头 / 值得抄作业”这类近域口语映射，V1 是否要先冻结一份有限白名单，而不是开放自由扩展？
   已冻结：需要先冻结有限白名单。白名单外表达不允许自由扩张映射，只能进入澄清、失败或保守降级语义。

3. 当一个请求既可能被解释为“项目推荐”，也可能被解释为“趋势方向推荐”时，V1 是默认给主解释并附次解释，还是必须用户先切换？
   已冻结：V1 默认给主解释，并可附带次解释；只有在高冲突且无法稳定判断时，才进入“需澄清”。

4. V1 是否允许把登录用户的 `interest profile` 作为推荐加权因素，还是先只做与用户无关的公共推荐语义？
   已冻结：不作为本次需求的阻塞项。V1 先只做与用户无关的公共推荐语义；登录态 `interest profile` 加权归入后续个性化能力建设，不进入当前需求冻结范围。
