# 自进化 Skill 沉淀与专属技能树需求分析

## 文档状态
- 状态：`Frozen for Design`
- 需求级别：`S3 - Large Feature / System Change`
- 需求来源：
  - 外部参考：[GenericAgent README（Chinese）](https://github.com/lsdefine/GenericAgent#chinese)
  - 仓库现状：`agentReadme.md`、`docs/specs/agent-work/*`、`docs/specs/product-specs/*`
- 本文目标：在不破坏现有 spec harness、review/preflight 门禁和 `Signal -> Filter -> Action` 主产品边界的前提下，冻结“任务自动沉淀 Skill，能力随使用持续增长，并形成专属技能树”的产品需求

## Requirement Size
`S3 - Large Feature / System Change`

判定原因：
- 这不是单点文档补充，而是新增一条长期生长型能力链路：`任务执行 -> 经验提炼 -> Skill 沉淀 -> 相似任务召回 -> 版本治理`
- 它会影响仓库级 Agent 工作方式、记忆分层、任务路由、治理策略和可观测性，而不只是某个命令或某一份产物
- 它需要同时约束“自动成长”和“工程纪律”，避免自进化机制反过来破坏现有 spec-first 开发模式

## 1. Background

### 需求来源
- 用户希望把 GenericAgent 的“自我进化机制”引入当前项目，使系统不再只依赖人工编写的固定 Skill，而是能把每次任务中的有效执行路径持续沉淀下来
- 外部参考强调的核心思想不是预置大量技能，而是“首次任务自主摸索，之后将执行路径固化为 Skill，并写入记忆层，在下一次同类任务中直接调用”
- 当前仓库已经有较强的 Agent 工作纪律，包括：
  - `agentReadme.md` 作为根上下文和强约束入口
  - `docs/specs/agent-work/*` 作为手工编写的 Skill / 流程书
  - `preflight`、`review`、`exec-plan`、`verification` 作为治理门禁
- 但当前仓库缺少“经验自动沉淀”的产品能力，导致可复用经验主要散落在聊天记录、exec-plan 账本、临时命令和人工记忆里

### 现状痛点
- 现有 Skill 主要依赖人工预先编写，新增能力的积累速度取决于维护者主动整理，而不是系统使用本身
- 同类任务重复出现时，Agent 不能稳定判断“这次是否已有可复用的仓库内经验”
- 任务完成后的有效经验缺少统一的沉淀标准，容易出现：
  - 经验只留在会话里
  - 经验只留在实现代码里
  - 经验只留在 exec-plan 的执行记录里
  - 经验虽存在，但没有被路由系统有效召回
- 当前仓库虽已有 `agent-work` 技能书和 `config.yaml` 中的 `memory` / `skills` 兴趣项，但尚未形成真正的“记忆分层 + Skill 生长 + 路由复用 + 失效治理”闭环

### 核心机会
- 当前仓库已经具备严格的 spec-first、review-first、verification-first 基础，适合承接“受控的自进化”
- 如果把自进化能力限定在仓库内 Agent 工作层，而不是直接侵入趋势雷达主产品裁决层，则可以同时获得：
  - 经验复利
  - 成本下降
  - 相似任务收敛更快
  - 不破坏主产品的可审计性和稳定性

## 2. Goals

### 产品目标
- 让当前仓库从“手工维护一套 Agent 工作技能书”升级为“人工 Skill + 任务后自动沉淀 Skill”的混合能力系统；其中人工 Skill 统一包含现有 Skill 书、后续人工新增 Skill，以及对自动生成 Skill 的人工修订版
- 让 Agent 在完成一个新任务后，能够把稳定、可复用、可验证的执行路径沉淀为后续可召回的 Skill 资产
- 让相似任务在下一次执行时，不必重新探索全部路径，而是能够优先命中已有 Skill，并在必要时继续增量修正
- 让沉淀出来的 Skill 不是无序列表，而是能够形成可浏览、可治理、可升级、可淘汰的“专属技能树”
- 让自进化机制始终服从当前仓库的 spec harness，不得绕过 `agentReadme.md`、`review`、`exec-plan`、`verification` 等既有纪律

### 用户价值
- 用户不再反复为相似任务支付相同的探索成本
- 仓库的 Agent 能力增长来自真实使用记录，而不只来自事前设计
- 维护者可以知道“当前这个仓库已经学会了什么、这些技能来自哪些任务、最近是否仍然有效”
- 技能增长变成可观测资产，而不是会话幻觉或不可审计的黑箱记忆

## 3. Users & Scenarios

### 用户角色
- 仓库所有者：希望 Agent 越用越熟悉本项目，减少重复指令和重复试错
- 编码 Agent：需要在执行新任务、相似任务、失败重试任务时获得更好的上下文与可复用流程
- 仓库维护者 / 审查者：需要确认自动沉淀的 Skill 不会绕过设计审批、代码审查和验证门禁

### 使用场景

#### 场景 1：第一次遇到某类任务
- 触发：用户提出当前仓库此前没有稳定处理过的新任务
- 目标：Agent 可以探索、试错、验证，并在任务完成后沉淀出候选 Skill，而不是把经验只留在聊天里

#### 场景 2：再次遇到相似任务
- 触发：用户提出与历史任务结构相近、目标相似或约束相近的任务
- 目标：系统优先召回已有 Skill，缩短探索路径，并明确告知使用了哪条经验

#### 场景 3：旧 Skill 失效或漂移
- 触发：仓库结构变化、脚本失效、依赖变化、规则更新，导致旧 Skill 不能直接复用
- 目标：系统能识别“旧经验命中但不再可靠”，进入修正而不是盲从，并把修正结果反写回 Skill 体系

#### 场景 4：技能库持续增长
- 触发：仓库经历数十次不同类型任务
- 目标：系统形成带有主题、父子关系、复用频次、最近验证状态的技能树，而不是堆积成难以维护的碎片

#### 场景 5：需要审计或治理
- 触发：维护者希望检查某条 Skill 的来源、适用范围、最后一次成功验证和当前状态
- 目标：系统能回答“这条 Skill 是从哪次任务学来的、适用于什么、最近是否还可信”

### 关键用户路径
1. 用户发起任务
2. 系统在执行前先进行相似任务检索与 Skill 路由
3. 如果命中稳定 Skill，则优先复用；若未命中，则按基础工作流执行
4. 任务结束后，系统从执行记录中提炼可复用经验
5. 符合条件的经验沉淀为 Skill 候选项
6. 候选 Skill 进入记忆层并可被后续任务召回
7. 复用次数、验证结果、失效情况持续更新，逐渐形成专属技能树

## 4. Requirement Definition

### 需求 1：自进化机制必须限定在仓库内 Agent 工作层，不能越界改写主产品裁决语义
- What：系统必须明确把“自进化 Skill”定义为仓库内 Agent 工作能力，而不是趋势雷达主产品的自动裁决器
- Why：当前仓库最核心的可信边界仍然是 `Signal -> Filter -> Action` 主产品链路与 spec harness；自进化机制只能增强 Agent 如何做事，不能直接改写产品事实判断
- Who：仓库所有者、维护者、编码 Agent
- When：任意 Skill 生成、召回、升级或淘汰时
- Expected outcome：用户获得更强的 Agent 工作能力，但趋势雷达主产品的可解释裁决边界不被污染

### 需求 2：每次完成任务后，系统必须具备“是否可沉淀为 Skill”的标准化判定
- What：任务结束后，系统必须判断这次执行是否产出了稳定、可复用、可验证的经验，并决定是否生成 Skill 候选；候选 Skill 与稳定 Skill 必须分层管理，不得一生成就默认进入高信任复用态
- Why：不是所有任务都值得沉淀；如果无门槛全量写入，会迅速造成记忆污染和技能碎片爆炸
- Who：编码 Agent、维护者
- When：任务完成并产生执行记录后
- Expected outcome：只有高价值、可复用、边界清晰的经验才进入 Skill 体系，且后续晋升路径可审计

### 需求 3：沉淀出的 Skill 必须是结构化资产，而不是一段聊天摘要
- What：Skill 必须至少表达任务意图、适用条件、输入约束、关键步骤、依赖前提、禁止越界、验证方式、失败信号和回退方式；V1 允许 Skill 挂接可执行脚本，但脚本只是 Skill 资产的一部分，不能替代结构化说明、来源记录和验证信息
- Why：只有结构化资产才可能被稳定路由、重复使用、审计和治理
- Who：编码 Agent、维护者
- When：任意候选 Skill 写入系统时
- Expected outcome：每一条 Skill 都能回答“什么时候能用、怎么用、依赖什么、用完如何验证、失效了怎么办”，并在存在脚本挂接时仍保留完整审计语义

### 需求 4：系统必须建立面向当前仓库的分层记忆体系
- What：系统必须区分不同层级的记忆，而不能把所有经验混在一个列表里
- Why：外部参考已经证明“任务 Skills / SOPs”只是长期能力的一层；当前仓库也已经天然存在元规则、项目事实、会话记录和技能书等不同信息层
- Who：编码 Agent、维护者
- When：设计自进化能力的存储、检索和路由时
- Expected outcome：系统能区分至少以下几类内容：
  - 元规则：`agentReadme.md`、仓库纪律、硬约束
  - 路由索引：用于相似任务召回的极简索引
  - 项目稳定事实：仓库结构、长期规则、稳定入口
  - 任务 Skill / SOP：可复用流程
  - 会话归档：具体任务执行痕迹和来源证明

### 需求 5：系统必须在任务开始前支持 Skill 检索与路由
- What：每次新任务开始时，系统必须能够判断是否存在可复用 Skill，并基于任务类型、目标、目标产物、约束、模块范围进行召回；任务类型与任务领域必须是可区分维度，不能用同一个粗粒度枚举重复表达；相似任务命中的产品原则必须冻结为“宁可漏召回，不可高置信误调用”，当命中依据不足时，系统必须显式回退为未命中，而不是强行套用旧 Skill
- Why：如果 Skill 只能“存起来”而不能“提前命中”，就不能形成能力复利
- Who：编码 Agent、仓库所有者
- When：每次接收新任务时
- Expected outcome：相似任务能够优先复用经验，而不是每次都从零探索；同时误命中不会因为默认强套旧经验而拖慢主任务

### 需求 6：Skill 复用失败时必须允许降级、修正和反写
- What：当已命中的 Skill 因仓库漂移、依赖变化、约束更新而失败时，系统必须允许回退到基础工作流，并把新的有效路径重新沉淀
- Why：自进化系统不能把旧经验当作绝对真理，否则会越来越僵化
- Who：编码 Agent、维护者
- When：Skill 被调用但验证失败时
- Expected outcome：系统具备“复用 -> 发现失效 -> 修正 -> 再沉淀”的闭环，而不是“命中旧 Skill -> 盲目执行 -> 继续失败”

### 需求 7：系统必须把沉淀结果组织成“专属技能树”，而不只是技能列表
- What：Skill 资产必须支持父子层级、主题聚类、来源任务链路、复用频次和状态视图
- Why：用户要的是“能力持续增长”，不是“越来越多的碎文件”；技能树强调继承关系、领域聚合和长期成长轨迹
- Who：仓库所有者、维护者
- When：Skill 数量增长后进行浏览、选择、治理时
- Expected outcome：用户能看到这个仓库在 `requirements/design/artifact-review（review）/testing/visual-console/automation` 等领域分别学会了什么

### 需求 8：自进化机制必须受现有治理体系约束，不能绕过审批与验证
- What：自动沉淀出来的 Skill 不能自动取得与现有人工 Skill 同等的信任级别，必须服从 review、preflight、verification 和设计审批边界；V1 只保留一个统一的人工 Skill 信任层，覆盖现有手工 Skill、后续人工新增 Skill，以及对自动生成 Skill 的人工修订版本
- Why：当前仓库的核心资产不是“会做事”，而是“在约束下做对事”；自动成长如果绕过治理，会直接破坏仓库可信度
- Who：维护者、编码 Agent
- When：Skill 候选生成、升级、默认启用、覆盖已有 Skill 时
- Expected outcome：自进化能力增强效率，但不削弱工程质量门禁

### 需求 9：系统必须支持 Skill 生命周期治理与失效复核
- What：Skill 必须具备候选、稳定、暂停、过期、废弃等状态，并记录升级、替换、冲突和淘汰原因；V1 允许候选 Skill 自动晋升为稳定 Skill，但自动晋升必须绑定显式信任条件，至少包括来源任务已完成验证、Skill 结构完整、未与根规则冲突，以及发生过后续成功复用或等价显式确认；对高频稳定 Skill，系统必须支持失效扫描 / 技能漂移检测
- Why：没有生命周期，技能库只会单向膨胀，最终失控
- Who：仓库所有者、维护者
- When：Skill 新建、复用、失败、升级、冲突时
- Expected outcome：用户能治理技能树，而不是被技能树反噬，并能知道哪些 Skill 正在增长信任、哪些 Skill 正在失效

### 需求 10：系统必须提供可观测的成长证据
- What：系统必须能够展示技能增长、命中率、复用成功率、失效率、最近验证时间和高价值领域覆盖情况
- Why：如果“能力在增长”不可观测，这套机制就只能停留在口号层
- Who：仓库所有者
- When：查看系统状态、评估机制效果、决定是否继续扩展时
- Expected outcome：用户可以判断这套自进化机制是否真的降低了重复劳动并提高了完成质量

## 5. Interpretation Options

### 关键请求 A：“把 GenericAgent 的自进化机制引导进项目”
- 解释 A1：把 GenericAgent 的整体运行范式、原子工具模型和本地计算机控制能力整体迁入本仓库
- 解释 A2：抽取 GenericAgent 的核心思想，只引入“任务后沉淀 Skill + 分层记忆 + 相似任务召回 + 技能树治理”这条能力线
- 推荐冻结：采用 `A2`
- 原因：当前仓库不是通用桌面代理，而是带有严格 spec harness 的趋势雷达工程仓库；全量迁入会导致边界错位

### 关键请求 B：“每次任务自动沉淀 Skill”
- 解释 B1：每个任务结束后无条件自动生成 Skill
- 解释 B2：每个任务结束后先生成候选，再按复用价值、边界清晰度和验证结果决定是否晋升为可复用 Skill
- 推荐冻结：采用 `B2`
- 原因：全量自动沉淀会快速制造噪声和污染；当前仓库更适合“自动提名，受控晋升”

### 关键请求 C：“形成专属技能树”
- 解释 C1：只要生成很多 Skill 文件，就视为形成技能树
- 解释 C2：技能树必须显式表达主题、父子关系、来源任务、状态和复用价值
- 推荐冻结：采用 `C2`
- 原因：列表不是树；树必须体现结构和成长关系

### 关键请求 D：“Skill 是什么”
- 解释 D1：Skill 只是 prompt 片段或自然语言备忘
- 解释 D2：Skill 是结构化 SOP 资产，可挂接脚本、验证命令、约束和回退规则
- 推荐冻结：采用 `D2`
- 原因：当前仓库强调可验证、可审计，纯 prompt 片段不足以承载长期工程经验

### 关键请求 E：“自动复用旧经验”
- 解释 E1：命中旧 Skill 后直接强制执行
- 解释 E2：命中旧 Skill 后优先参考与尝试，失败时允许降级并纠偏
- 推荐冻结：采用 `E2`
- 原因：旧 Skill 可能因仓库变化失效；强制执行会放大漂移风险

## 6. Constraints

### 权限与角色边界
- 自进化 Skill 不能覆盖 `agentReadme.md` 的根规则地位
- 自进化 Skill 不能绕过“设计未批准不得创建对应 exec-plan”的既有规则
- 自进化 Skill 不能绕过 review、preflight、verification 门禁
- 自进化 Skill 不能把未验证的经验伪装成仓库稳定事实
- 自进化 Skill 不能仅凭单次生成结果自动取得稳定高信任身份

### 数据与归属约束
- 人工 Skill、learned Skill、项目稳定事实、任务经验、会话归档必须可区分
- 任务经验、Skill 候选、稳定 Skill、会话归档必须可区分
- Skill 必须保留来源任务和验证痕迹，不能只保留脱离上下文的结论
- 项目稳定事实与一次性执行技巧必须分层存放，不能互相污染
- 挂接脚本、结构化说明、验证记录和来源任务必须可相互追溯，不能只留下脚本文件而失去语义归属
- 供路由与治理使用的 canonical 记录，与 `tree/*`、清单、总览等派生产物必须可区分；后者允许重建，不得成为唯一真相

### 工程纪律约束
- 该能力必须增强现有 `agent-work` 体系，而不是替换它
- 现有人工 Skill 书、后续人工新增 Skill 和人工修订版 Skill 仍然是 V1 的重要信任来源，不得被自动生成内容静默覆盖
- 自进化机制引入后，现有仓库命令、主产品流水线和 spec harness 必须继续可运行
- 纯文件系统是 V1 的技能树查询与管理入口，不以 CLI 或 visual console 作为冻结前提

### 用户可见语义约束
- 系统必须允许用户区分“人工冻结的规范”与“自动沉淀的经验”
- 系统必须允许用户区分“候选 Skill”与“稳定 Skill”
- 系统必须允许用户知道“本次任务用了哪些已有 Skill，哪些地方重新探索了”
- 当存在多个候选 Skill 但最终未命中时，系统必须允许表达“未找到足够可信的复用路径”，而不是伪装成精准命中

### 安全与质量约束
- 技能自动沉淀不能鼓励把一次性的危险命令、临时硬编码、环境偶然性写成可复用默认方案
- 技能复用必须保留失败信号和回退路径，不能只有成功路径
- 技能树增长不能以牺牲审计可读性和仓库可维护性为代价
- 高频稳定 Skill 不能被默认为永久有效，必须允许周期性失效扫描与漂移降级

## 7. Acceptance Criteria

### 验收 1：首次任务完成后可产生候选 Skill
- 可观测结果：系统能从一次成功完成且已验证的任务中生成候选 Skill 和记忆归档
- 成功条件：候选 Skill 明确表达任务类型、适用范围、关键步骤、验证方式和来源任务；若挂接脚本，也能明确脚本与 Skill 元信息的对应关系
- 失败条件：经验只留在聊天或 exec-plan 记录里，后续无法被系统识别和复用

### 验收 2：相似任务开始前可命中相关 Skill
- 可观测结果：系统在任务开始阶段显式给出命中的 Skill 或说明未命中
- 成功条件：相似任务能够优先复用已有经验，减少重复探索；当命中依据不足时，系统明确选择未命中而不是高置信误调用
- 失败条件：Skill 虽然已沉淀，但后续任务仍然完全从零开始，或系统频繁误命中不适用的旧 Skill 导致路径更差

### 验收 3：Skill 复用失败时可降级并纠偏
- 可观测结果：当旧 Skill 失效时，系统会回退到基础工作流，并记录失效原因与修正结果
- 成功条件：失败不会导致系统僵死，且修正经验可重新进入沉淀流程
- 失败条件：系统盲从旧 Skill，持续复现同类错误

### 验收 4：技能树是结构化可浏览资产
- 可观测结果：用户可以按领域、层级、状态、最近验证时间和复用频次查看技能资产
- 成功条件：技能库不只是文件列表，而是能在纯文件系统入口下表达成长关系和治理状态
- 失败条件：技能越来越多，但用户无法知道该用哪个、哪些已过期、哪些最可靠

### 验收 5：自进化机制不破坏现有治理边界
- 可观测结果：自动沉淀出来的 Skill 仍受现有 review / preflight / verification / approval 纪律约束
- 成功条件：仓库质量门禁继续成立；自动晋升、脚本挂接和复用命中都不会绕过既有治理
- 失败条件：自动 Skill 绕过治理，造成规范漂移、实现漂移或验证失真

### 验收 6：成长效果可被衡量
- 可观测结果：系统能够展示技能数量、有效命中率、复用成功率、失效率和高价值领域覆盖情况
- 成功条件：用户可以客观判断“这套机制是否真的越用越强”
- 失败条件：系统宣称自进化，但没有任何可验证的成长证据

### 验收 7：高频稳定 Skill 可被失效复核
- 可观测结果：系统能够对高频稳定 Skill 执行失效扫描或漂移检测，并输出是否继续可信
- 成功条件：高频 Skill 在仓库环境变化后不会长期以“默认稳定”身份潜伏失效
- 失败条件：系统从不复核高频 Skill，导致过时能力长期高信任暴露

## 8. Risks & Gaps

### 风险
- 记忆污染：把一次性方案、脏命令、环境偶然性沉淀为长期 Skill
- 技能碎片化：技能过细、过窄、过多，导致路由成本反而上升
- 过度泛化：把只适用于一个局部场景的经验误判为通用 Skill
- 规范漂移：自动生成经验与仓库冻结规则发生冲突
- 盲目信任旧经验：仓库已变化，但系统仍高置信复用过时 Skill
- 治理成本失控：技能数量上升后，维护成本超过收益
- 脚本挂接异化：用户只看到脚本可执行，却无法理解其适用边界与失败条件
- 多 Skill 竞争：一个任务同时命中多条旧经验，但系统缺少保守降级口径，反而放大误路由

### 残余缺口与补齐策略

#### 缺口 1：多 Skill 同时命中时的用户可见语义尚未完全冻结
补齐策略：
- 设计阶段需明确“主命中 / 辅助参考 / 全部放弃复用”的表达方式
- 必须保持“宁可漏召回，不可高置信误调用”的保守原则

#### 缺口 2：脚本型 Skill 与纯文档型 Skill 的统一治理视图尚未完全冻结
补齐策略：
- 设计阶段需明确两类 Skill 在文件系统中的并列关系、命名关系和审计入口
- 不允许脚本挂接破坏 Skill 结构化信息的可读性

#### 缺口 3：高频 Skill 的失效扫描节奏与触发条件尚未完全冻结
补齐策略：
- 设计阶段需明确“定期复核”与“事件触发复核”的边界
- 必须保证高频稳定 Skill 存在可见的继续可信 / 待复核 / 已降级状态

#### 缺口 4：技能树初始分类已冻结方向，但层级粒度仍需设计阶段细化
补齐策略：
- V1 以当前仓库真实任务域作为初始分类基础，如 `requirements / design / exec-plan / code-implementation / artifact-review / testing / visual-console / automation / repo-ops`
- 设计阶段只允许在此基础上细化，不允许重回无边界自由分类

## 9. Open Questions

当前无阻塞性开放问题。

本轮已冻结的关键结论：
1. V1 允许候选 Skill 自动晋升为稳定 Skill，但必须满足显式信任条件，不能无门槛自动晋升。
2. V1 允许 Skill 挂接可执行脚本，但脚本不能替代结构化说明、来源记录和验证信息。
3. V1 的技能树查询与管理入口冻结为纯文件系统；CLI 与 visual console 不是本轮需求前提。
4. 相似任务命中的冻结原则是“宁可漏召回，不可高置信误调用”；命中依据不足时必须显式未命中。
5. V1 需要显式引入“失效扫描 / 技能漂移检测”能力，用于定期或事件触发复核高频 Skill。

以上结论已回写到需求定义、约束、验收与风险章节，设计阶段不应再把这些问题当作待定前提。

`READY`
