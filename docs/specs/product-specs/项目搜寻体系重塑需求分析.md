# 项目搜寻体系重塑需求分析

## 文档状态
- 状态：`Draft for Review`
- 需求级别：`S3 - Large Feature / System Change`
- 分析方法：依据[需求分析_skill.md](需求分析_skill.md#L1)输出
- 本文目标：在不直接进入方案设计与实现拆解的前提下，冻结“项目搜寻体系重塑”的产品需求边界，解决当前项目覆盖率不足、头部项目长期霸榜、每日结果重复度高，以及用户真实需求方向搜不到而导致的转化与留存问题。

## Requirement Size
`S3 - Large Feature / System Change`

判定原因：
- 这不是单点权重调参，而是覆盖 `搜寻 -> 候选 -> 理解 -> 排序 -> 分发 -> 反馈` 的系统级能力重塑。
- 该需求同时影响候选来源、主题覆盖、结果表达、新鲜度、多样性和转化路径。
- 该需求跨越现有 `signal / normalize / filter / action` 的产品语义边界，并会改变用户对 daily 结果“为什么值得看”的理解方式。

## 1. Background

### 需求来源
- 用户明确反馈：当前系统漏掉了他们实际想找的方向，例如“智能导购 agent”“股票相关 agent”等，导致搜不到感兴趣项目并产生流失。
- 当前头部项目如 `Claude`、`Hermes`、`OpenClaw` 一类对象，因为持续高 star 增长和多信号命中，长期占据前排，压缩了其他项目的展示机会。
- 现有 daily 前排结果重复度偏高，用户容易形成“每天都在看同一批项目”的感受，削弱继续打开和持续浏览的动机。

### 业务背景
- 当前系统的核心价值仍是“从高噪声生态里找出值得看的 agent 相关项目”，而不是泛化成任意技术资讯聚合器。
- 当前系统已经具备可解释评分、freshness 区分、候选归一化与日报输出链路，但搜寻范围仍偏向少量通用热度源与预定义生态词。
- 当前系统更擅长回答“今天全局热什么”，但还不够擅长回答“用户此刻想找什么”以及“除了熟知头部项目外还有什么值得看”。

### 用户动机
- 用户希望看到更多“自己真的会点开”的项目，而不是反复看到已经耳熟能详的头部项目。
- 用户希望系统既能覆盖全局热点，也能覆盖垂直需求方向和新出现的中腰部项目。
- 用户希望系统不是简单把更多项目塞进列表，而是能更稳定地给出“相关、新鲜、可探索”的结果组合。

## 2. Goals

### 产品目标
- 提升项目搜寻体系的覆盖率，让系统能够发现更多用户真实关心的 agent 方向和更广行业软件智能化项目。
- 提升结果的新鲜度与变化感，避免少数持续强势的头部项目长期挤占前排。
- 提升转化与留存，让用户更容易在一次浏览中看到“我想看的项目”，并愿意继续浏览、收藏、回访。
- 保持产品的可解释性，让用户理解项目为什么出现、为什么没出现、为什么排在这里。

### 用户价值
- 用户能看到更多此前系统经常漏掉的垂直方向项目。
- 用户能更频繁遇到“没见过但值得看”的新项目，而不是连续多天重复的熟悉项目。
- 用户能更快判断某个结果是“全局热点”还是“和自己的需求相关”，从而减少无效浏览。

## 3. Users & Scenarios

### 用户角色
- 趋势扫描型用户：想快速知道今天 agent 生态里有哪些值得关注的新项目与新方向。
- 任务导向型用户：带着明确目标来找项目，例如智能导购 agent、股票 agent、浏览器 agent、coding agent。
- 持续回访型用户：每天或隔几天回来，希望看到变化，而不是同样的前排列表。
- 研究与决策型用户：关注的不只是热度，还包括覆盖面、方向代表性和结果解释性。

### 使用场景

#### 场景 1：用户做日常扫描
- 触发：用户打开 daily 结果，希望快速了解今天有哪些值得看的项目。
- 目标：在有限时间内看到“新鲜的、不同的、值得点开”的项目组合，而不是继续被少数熟悉项目占满前排。

#### 场景 2：用户带着明确需求找项目
- 触发：用户想找某一类 agent，例如智能导购、股票研究、金融分析、行业工作流 agent。
- 目标：系统至少应有能力捕捉到该方向的候选，或明确表达“当前未发现足够信号”，而不是完全无覆盖。

#### 场景 3：用户多日回访
- 触发：用户连续多天打开产品。
- 目标：用户感知到榜单存在真实变化，且新项目、垂直项目、中腰部项目能够获得稳定展示，而不是永远只看到同一批头部项目。

#### 场景 4：用户想判断某个项目为什么值得看
- 触发：用户在结果里看到一个项目，想知道它为何入选，它代表的是全局热点还是需求相关结果。
- 目标：系统能解释其入选来源、价值类型和曝光理由，帮助用户决定是否点开。

### 触发条件
- 新的一次 daily 搜寻与筛选开始时。
- 用户显式或隐式表现出对某些垂直方向的关注时。
- 系统发现当前前排结果与近期结果高度重复时。
- 系统观察到搜寻覆盖与用户需求之间存在缺口时。

### 关键用户路径
1. 系统从多类来源获得候选项目与方向信号。
2. 系统将候选区分为全局热点、需求相关候选、持续强势头部项目、可探索新项目等不同对象。
3. 系统根据相关性、新鲜度、重复控制与可信度形成可消费结果。
4. 用户既能看到今日全局值得看的热点，也能看到更贴近需求或更有新鲜感的项目。
5. 用户的点击、跳过、搜索、收藏、订阅、显式反馈等行为继续反哺后续搜寻范围与展示策略。

## 4. Requirement Definition

### 需求 1：系统必须把“搜得更广”定义为“方向覆盖 + 需求覆盖”，而不只是增加抓取量
- What：系统必须同时覆盖全局热门方向、既有生态方向以及用户真实需求驱动的垂直方向；覆盖率不能仅通过增加候选总数来定义。
- Why：仅扩大候选数量并不能解决“用户想找的东西根本没进池”的问题。
- Who：趋势扫描型用户、任务导向型用户、持续回访型用户。
- When：每次项目搜寻与候选收集时。
- Expected outcome：用户想找的方向更有机会被系统纳入，而不是仍然只能看到通用热门项目。

### 需求 2：系统必须把“全局热度”和“用户想看”明确区分为两类不同语义
- What：系统必须能够区分“全局热点项目”和“需求相关项目”，两者都可以被展示，但不能混成单一无解释榜单。
- Why：用户会同时存在“看看今天全网热什么”和“找我此刻需要什么”两种目标。
- Who：所有用户。
- When：排序、曝光与结果表达时。
- Expected outcome：用户能理解某个项目是因为全局热度进入视野，还是因为与特定需求方向相关而进入视野。

### 需求 3：系统必须限制持续强势头部项目对前排曝光的长期挤占
- What：系统必须识别“持续强势但已被反复曝光”的项目，并在缺乏新增价值或连续多日重复出现时，降低其前排优先级。
- Why：否则少数项目会因强者恒强而长期压制新项目与垂直项目，损害新鲜度和探索价值。
- Who：持续回访型用户、趋势扫描型用户。
- When：生成 daily 前排结果与核心曝光位时。
- Expected outcome：头部项目仍可被看见，但不再无限期占据首要曝光位置；其他项目有稳定进入前排的机会。

### 需求 4：系统必须将“多样性”定义为用户可感知的结果差异，而不是单独结果类别
- What：系统必须让用户在一次浏览和跨日浏览中，感知到主题差异、成熟度差异、热度层级差异与新旧差异；但 V1 不单独设立“多样性结果”类别。
- Why：如果多样性只停留在内部标签层，或被做成脱离用户语义的独立桶，用户仍会感知为“每天看见的是同样的东西”。
- Who：持续回访型用户、趋势扫描型用户。
- When：前排组合、列表分层与摘要表达时。
- Expected outcome：用户明显感知到结果里存在不同类型的值得看对象，而不是只有一类熟悉头部项目。

### 需求 5：系统必须支持需求驱动的垂直方向发现
- What：系统必须支持对垂直 agent 方向和更广行业软件智能化方向进行正式搜寻与消费，例如智能导购、股票研究、金融分析、行业工作流 agent 等，而不是只关注通用基础设施。
- Why：用户流失的重要原因之一，是具体需求方向没有被纳入可搜寻、可发现范围。
- Who：任务导向型用户。
- When：用户带目的地寻找项目，或系统进行需求导向覆盖时。
- Expected outcome：用户在这些方向上能看到候选、观察对象，或明确的“当前缺乏信号”状态，而不是完全无响应。

### 需求 6：系统必须让“为什么出现”与“为什么没出现”都可解释
- What：系统必须能解释项目入选原因、曝光原因、未入选原因或未被提升原因。
- Why：随着搜寻体系变复杂，如果没有解释层，用户会把遗漏、重复和排序变化理解为黑盒随机波动。
- Who：所有用户，尤其是研究与决策型用户。
- When：结果展示、下钻查看或对比判断时。
- Expected outcome：用户能理解项目与方向的进入逻辑，减少对系统不稳定或不可信的感受。

### 需求 7：系统必须建立未满足需求的反馈闭环
- What：当用户想看的方向长期搜不到、点击后快速离开、反复跳过头部项目时，系统必须把这些行为视作需求缺口信号，而不是单纯流量波动。
- Why：如果系统不从未命中与未转化行为中学习，覆盖率和留存问题会持续重复。
- Who：产品方、所有终端用户。
- When：用户浏览、点击、搜索、收藏、订阅、忽略和回访后。
- Expected outcome：系统后续搜寻与曝光范围能逐步贴近真实需求，而不是长期停留在静态配置。

### 需求 8：系统必须在扩展覆盖后仍保持可审计和可降级
- What：即使引入更多方向、更多来源和更多用户需求信号，系统也必须保持边界清晰、判断可解释、异常时可回退。
- Why：覆盖率扩张如果破坏稳定性，会直接损害用户信任。
- Who：所有用户。
- When：任何新来源、新方向、新反馈信号进入正式判断链路时。
- Expected outcome：系统既更广、更活，也不会因为扩张而变成不可解释的黑盒推荐流。

## 5. Interpretation Options

### 关键诉求 A：“提升覆盖率”
- 解释 A1：尽可能抓更多项目，只要候选池总数变多就算覆盖率提升。
- 解释 A2：覆盖率必须同时包含“方向覆盖”和“需求覆盖”，用户想找的方向能否进池比总量更重要。
- 推荐冻结：采用 `A2`
- 原因：如果只追求总量增长，系统仍可能继续漏掉智能导购、股票 agent 这类具体需求方向。

### 关键诉求 B：“减少前排重复”
- 解释 B1：从底层候选池里直接压制或删除头部项目。
- 解释 B2：保留头部项目的观察与判断价值，但在曝光层限制其长期挤占前排。
- 推荐冻结：采用 `B2`
- 原因：头部项目不应被伪装成“不存在”，真正要控制的是它们对核心曝光位的长期垄断。

### 关键诉求 C：“给用户更想看的项目”
- 解释 C1：只做一套全局榜单，默认用户都看同一种“最热”。
- 解释 C2：在保留全局热点价值的同时，正式承认“需求相关”是另一条同等重要的产品语义。
- 推荐冻结：采用 `C2`
- 原因：用户既有全局扫描需求，也有具体找项目需求，两者不能继续被强行压缩成一个单榜语义。

### 关键诉求 D：“纳入智能导购 agent、股票 agent”
- 解释 D1：把搜寻边界扩张到任意热门软件与行业工具。
- 解释 D2：扩张到 agent 化、自动化、智能决策型的垂直场景项目，并进一步覆盖更广的行业软件智能化项目。
- 推荐冻结：采用 `D2`
- 原因：产品核心仍是 agent trend radar，但 V1 的覆盖边界需要比当前显著更广。

### 关键诉求 E：“提升转化与留存”
- 解释 E1：只优化排序，让更热项目排得更靠前。
- 解释 E2：同时优化覆盖、前排去垄断、结果语义分层、无结果表达和未满足需求反馈闭环。
- 推荐冻结：采用 `E2`
- 原因：转化与留存问题不只是排序问题，本质上是“结果池不对、重复太高、解释不清、需求未被学习”的综合问题。

## 6. Constraints

### 权限与裁剪边界
- 系统不能因为追求覆盖率而放弃“agent 相关项目”这一核心边界。
- 用户需求信号可以影响搜寻与曝光，但不能把系统直接改写成纯个性化黑盒推荐流。
- 全局热点语义必须保留，不能被完全替换为需求结果。
- 任何“压制头部项目”的机制都不能伪造事实，不得把高热项目从真实观察层中抹去。

### 数据与事实边界
- “搜到更多”不等于“值得展示更多”；覆盖率与最终曝光必须区分。
- “用户想看”不等于“全局更重要”；个性相关性与客观热度必须分层表达。
- “重复度高”必须按用户可见结果定义，而不是只按内部 repo 去重定义。
- 对垂直方向的覆盖必须能够落回明确项目、明确方向，或明确的“当前缺乏信号”状态，不能伪造结果。

### 业务规则约束
- 系统必须同时回答三类问题：
  - 今天全局值得看的是什么
  - 今天有哪些新鲜或不重复的东西值得看
  - 对于特定需求方向，有哪些候选值得看
- 在同一次 daily 消费中，系统允许且应当允许“全局热点”和“需求相关结果”获得独立表达位置，而不是强制压缩进单一热榜排位逻辑。
- V1 不单独设置“多样性结果”这一正式结果类别；多样性通过曝光约束实现。
- daily 核心曝光位中，全局热点应固定占据约一半席位，作为用户理解当日生态变化的主锚点。
- 核心曝光位剩余席位，应优先分配给需求相关结果，以及非头部、非陈词滥调、具备新鲜感的中腰部或新项目。
- 对连续多日重复曝光、缺乏新增价值、且已具备高度公众认知的头部项目，应执行优先级后移，为其他项目释放展示机会。
- 任何扩展后的搜寻体系都必须支持“持续回访时的结果变化感”。
- 用户流失相关反馈必须能够进入后续需求覆盖逻辑，而不是停留在人工口头总结。

### 兼容性约束
- 当新增方向、来源或反馈信号不可用时，系统仍必须维持现有主链路产出。
- 不得因为引入新搜寻语义而让现有 daily/weekly 完全失去可比较性。
- 扩展后系统仍必须允许区分“规则主判断”和“需求/曝光增强判断”。

## 7. Acceptance Criteria

### 验收 1：覆盖率不再只表现为总量增加
- 可观察结果：结果中能持续出现此前常被漏掉的垂直 agent 方向项目，或对缺失方向给出明确的“暂未发现足够信号”表达。
- 成功条件：用户提出的具体需求方向不再长期处于完全无覆盖状态。
- 失败条件：候选总数增加了，但用户关心的方向仍旧搜不到。

### 验收 2：前排重复度显著下降
- 可观察结果：连续多日浏览时，核心曝光位不再长期被同一批项目固定占据。
- 成功条件：头部项目仍可见，但用户明显感知到前排存在真实变化和新对象进入。
- 失败条件：虽然系统内部做了更多处理，用户看到的前排仍几乎没有变化。

### 验收 3：多样性是用户可感知的
- 可观察结果：一次浏览中，用户能同时看到全局热点、垂直需求命中、新观察对象或其他不同类型结果。
- 成功条件：用户不再把结果感知为“同类热门项目的重复堆叠”。
- 失败条件：内部标签更丰富了，但用户视角仍觉得都是同一类项目。

### 验收 4：任务导向用户能更容易命中所需项目
- 可观察结果：带着明确需求进入的用户更容易看到与需求方向相关的候选或观察结果。
- 成功条件：像智能导购 agent、股票 agent 这类方向不再长期空白。
- 失败条件：系统仍只能返回一批泛化热门项目，与具体需求无明显关系。

### 验收 5：结果语义可解释
- 可观察结果：用户能理解某个项目是因为全局热点、垂直需求匹配、新鲜观察还是持续跟踪而出现。
- 成功条件：用户能回答“它为什么在这里”和“为什么别的项目没在这里”。
- 失败条件：系统扩张后结果更像黑盒推荐，解释能力反而下降。

### 验收 5A：需求相关结果拥有独立表达位置
- 可观察结果：在同一次 daily 消费中，用户可区分“今日全局热点”“与你需求相关”等不同语义区域，或其等价表达。
- 成功条件：需求相关结果不会因为全局热榜排位过强而完全失去进入主视野的机会。
- 失败条件：虽然系统内部已识别需求相关结果，但最终展示仍退化为单一热榜，用户看不到独立表达。

### 验收 5B：核心曝光位不会被头部熟知项目长期垄断
- 可观察结果：daily 核心曝光位中，全局热点稳定占据约一半席位；剩余席位不会长期被 `Claude`、`Codex` 等高认知头部项目反复占满。
- 成功条件：系统能够在保留全局热点观察价值的同时，稳定让出展示空间给需求相关结果和非头部的新鲜项目。
- 失败条件：即使引入新搜寻能力后，用户看到的前排仍主要是熟知头部项目的重复出现，新的中腰部或垂直方向项目难以进入主视野。

### 验收 6：未满足需求能形成反馈闭环
- 可观察结果：用户持续搜不到、跳过、快速离开的方向会在后续成为可识别的需求缺口信号。
- 成功条件：系统的后续搜寻与曝光语义能根据真实需求缺口发生调整。
- 失败条件：用户反馈只停留在人工总结，系统行为没有持续学习。

### 验收 7：扩张后系统仍可降级
- 可观察结果：新增来源、反馈信号或曝光约束失效时，系统仍能稳定产出基础结果。
- 成功条件：系统不会因为新能力异常而让主产物失真或不可用。
- 失败条件：扩张能力一旦异常，主结果就出现大面积失真或断裂。

## 8. Risks & Gaps

### 风险
- 如果只追求覆盖率，系统可能演变为高噪声项目堆积，转化率反而下降。
- 如果只压制头部项目而不保留其真实价值，系统会损害“全局热点观察”能力。
- 如果把需求相关性和全局热度混写，用户会无法理解榜单含义。
- 如果不为需求相关结果保留独立表达位置，系统后续即使具备更强召回能力，最终仍会退化为单一热榜逻辑。
- 如果多样性没有用户可见定义，只做内部去重，产品层面感知不会改善。
- 如果垂直方向边界不冻结，系统可能失控扩张为泛行业信息平台。

### 缺口
- “明确用户意图”的最小判定口径属于个性化记忆与后续行为建模范围，不纳入本次项目搜寻体系重塑需求冻结。
- V1 已冻结核心曝光份额原则：全局热点约占一半席位，其余席位由需求相关结果与非头部新鲜项目竞争获取；更细颗粒度的数值分配留待设计阶段细化。

### 隐含假设
- 用户愿意接受“全局热度”和“需求相关”是两类不同但都重要的结果。
- 用户更在意“能看到想看的项目”和“每天有变化感”，而不只是榜首是否仍是最热项目。
- 产品后续会允许把用户行为反馈作为正式产品信号，而不是永远停留在静态配置。

## 9. Open Questions

以下问题已结合当前产品判断完成冻结，可作为设计阶段输入：

1. V1 正式覆盖的垂直方向范围要冻结到什么边界？
   已冻结：进一步覆盖更广的行业软件智能化项目，而不是只停留在 agent 化工作流与智能决策场景的窄边界。

2. “需求相关结果”是否允许在同一次 daily 消费中获得独立表达位置？
   已冻结：允许，而且必须允许。否则系统最终仍会退化为单一热榜逻辑，需求相关召回即使存在也会被全局热度吞没。

3. 对持续强势头部项目的曝光限制应以什么产品语义定义？
   已冻结：`连续多日重复曝光` 与 `缺乏新增价值` 两个条件任一成立时，都要触发优先级后移；两者都属于头部项目让位机制。其目标不是删除头部项目，而是避免其长期占据核心曝光位，给其他项目展示机会。对于 `Codex`、`Claude` 这类高度知名项目，默认不必长期靠前展示。

4. 未满足需求反馈的最小输入范围是什么？
   已冻结：除点击、跳过、回访之外，也包含搜索、收藏、订阅、显式反馈。

5. 当某个垂直方向暂时没有足够信号时，产品是否需要正式输出“无结果但已搜索”的语义？
   已冻结：需要输出。否则用户会把无覆盖误解为系统没有该能力。

6. 全局热点、头部去垄断约束、需求相关结果三者的优先级关系是否需要在 V1 就冻结？
   已冻结：需要在 V1 冻结，且采用“先质量门槛，后曝光约束”的两层规则。
   - 第一层：任何结果必须先通过基础质量门槛，不过门槛则不进入正式核心曝光。
   - 第二层：通过门槛后，核心曝光位中全局热点固定占据约一半席位。
   - 剩余席位优先留给需求相关结果，以及非头部、非陈词滥调、具备新鲜价值的中腰部或新项目。
   - V1 不设独立的“多样性结果”类别；多样性通过“头部去垄断 + 新鲜项目获得稳定露出”的曝光约束来实现。
   - 对 `Claude`、`Codex` 一类高度熟知且连续多日重复出现、或缺乏新增价值的项目，执行优先级后移，不长期占据前排。

## 10. Requirement Freeze Status

`READY`

原因：
- 核心问题、用户价值、需求边界和验收方向已经清晰。
- 已冻结的核心原则包括：
  - 覆盖率必须同时包含方向覆盖与需求覆盖
  - 头部项目问题本质上是曝光垄断问题，不是事实删除问题
  - 多样性不作为独立结果类别，而作为曝光约束目标存在
  - 系统必须正式支持垂直 agent 与更广行业软件智能化方向发现
  - 全局热点与需求相关结果必须语义分层

  - 头部项目在“连续多日重复曝光”或“缺乏新增价值”时应后移优先级

- “明确用户意图”的最小判定口径已明确不属于本次需求冻结范围，而属于后续个性化记忆能力建设。
